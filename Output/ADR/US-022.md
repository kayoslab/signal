## ADR-022: Permission Debt Score Calculation Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-022

### Context

The application needs a deterministic score (0–100) representing the cumulative risk of granted browser permissions, as part of the Permission Debt dashboard module (CLAUDE.md Required Module #4). The `src/scoring/` directory exists in the repository structure (per CLAUDE.md) but contains only a `.gitkeep` placeholder. The `src/permissions/permissions-adapter.ts` (US-021) already provides a `PermissionCheckResult` interface and `PermissionState` type that this module will consume. The signal collectors (US-006 through US-009) and permissions adapter (US-021) establish clear patterns of typed interfaces, pure functions, and safe fallbacks.

### Decision

Add a permission debt scoring module (`src/scoring/permission-debt-score.ts`) that:

1. Defines a `PermissionWeight` record mapping each of the 11 standard permissions to a risk weight, with privacy-invasive permissions (camera, microphone, geolocation) weighted higher than motion sensors (accelerometer, gyroscope, magnetometer)
2. Exports a pure function `calculatePermissionDebtScore(permissions: PermissionCheckResult[]): number` that sums weights only for `granted` states, normalizes to 0–100 (sum of granted weights / sum of all input weights × 100), and returns a deterministic integer
3. Exports a `PermissionDebtResult` interface containing `score`, `maxPossible`, and per-permission breakdown
4. Includes a default weight fallback for unknown permission names, ensuring forward compatibility if new permissions are added to the adapter
5. Documents scoring rules, weight rationale, and normalization formula via JSDoc
6. Unit tests in `tests/us-022/` covering all scoring scenarios including boundary conditions, state filtering, determinism, and graceful handling of edge cases

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to an existing empty directory (`src/scoring/`) and one new test directory (`tests/us-022/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports only from the existing `src/permissions/permissions-adapter.ts` types. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover the new files without modification.

**Risks identified and mitigations:**
- **Weight assignments are subjective:** The risk weights for each permission are judgment calls (e.g., camera higher than accelerometer). The plan correctly requires JSDoc documentation of the rationale for each weight, making the reasoning transparent and auditable for downstream consumers (US-023 Permission Debt UI, US-027, US-029). Downstream tickets can adjust weights if needed.
- **Fixed permission set assumption:** If the permissions adapter's `STANDARD_PERMISSIONS` list changes, the same set of granted permissions would yield a different score. The plan mitigates this by deriving `maxPossible` from the actual input array (summing weights of all provided permissions) rather than a hardcoded constant. This is the correct design — the score is relative to the permissions actually queried.
- **Unknown permission names:** If the adapter adds new permission names not present in the weight table, the plan includes a default weight fallback. This prevents silently ignoring new permissions and ensures the score remains meaningful as the adapter evolves.
- **Integer rounding:** The normalization formula produces a float that must be rounded to an integer. The plan should use `Math.round()` for the final value. This is an implementation detail, not an architectural concern.
- **Empty input array:** The plan covers this edge case — an empty array returns score 0. The normalization formula would produce 0/0; the implementation must guard against division by zero.

**Consistency with existing architecture:** Fully consistent. This is the first module in `src/scoring/`, establishing the scoring pattern for future scoring modules. It follows CLAUDE.md conventions: pure functions, data collection (permissions adapter) separated from scoring logic, heuristics are explainable (JSDoc-documented weights), no DOM logic in scoring modules. The module imports types from `src/permissions/` — a clean cross-directory dependency that follows the repository's layered structure (signals → permissions → scoring → modules). Tests land in `tests/us-022/` following the per-ticket convention. No runtime dependencies added. All computation is client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Establishes the first scoring module and a reusable pattern (weight table + pure scoring function + result interface) for potential future scoring needs
- **Positive:** Deterministic, pure-function design makes the score trivially testable and reproducible
- **Positive:** Input-relative normalization (max derived from input, not hardcoded) ensures the score remains valid as the permission set evolves
- **Positive:** Default weight fallback provides forward compatibility without requiring synchronized updates between adapter and scorer
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All computation remains client-side; no trust rule violations
- **Negative:** Weight assignments are inherently subjective — clearly documented rationale mitigates but does not eliminate this concern
- **Trade-off:** The 0–100 normalization means adding or removing permissions from the adapter changes the score for the same set of granted permissions; this is acceptable because the score represents relative debt within the queried permission set, not an absolute measure
