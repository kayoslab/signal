## ADR-023: Permission Debt Dashboard Module

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-023

### Context

The application needs a Permission Debt dashboard module (CLAUDE.md Required Module #4) that displays current permission states and cumulative risk. The `src/permissions/permissions-adapter.ts` (US-021) exports `checkPermissions(): Promise<PermissionCheckResult[]>` returning results for 11 standard permissions with states `'granted' | 'denied' | 'prompt' | 'unsupported'`. The `src/scoring/permission-debt-score.ts` (US-022) exports `calculatePermissionDebtScore(permissions): PermissionDebtResult` with `score` (0–100), `maxPossible`, and `breakdown` fields. The `src/modules/` directory contains two established modules — `intro/` (3 files + CSS) and `fingerprint/` (3 files, no co-located CSS) — establishing the convention of per-module subdirectories with barrel exports, render functions, and format helpers. The `src/ui/receipt.ts` (US-011) provides `createReceipt(title, rows)` for structured label/value display. The `src/app/main.ts` currently renders intro then shell, with no modules yet mounted to the `#dashboard` container.

### Decision

Add a Permission Debt module with:

1. **Render function** (`src/modules/permission-debt/permission-debt-module.ts`) — async `renderPermissionDebtModule(): Promise<HTMLElement>` that calls `checkPermissions()`, passes results to `calculatePermissionDebtScore()`, formats permission rows with state badges, displays the debt score prominently, and adds explanatory text about what permission debt means
2. **Format helper** (`src/modules/permission-debt/format-permissions.ts`) — maps `PermissionCheckResult[]` to display rows with human-readable labels, state-based color coding using semantic design tokens, and filtering to only show supported permissions per ticket notes
3. **Scoped styles** (`src/modules/permission-debt/permission-debt.css`) — module-specific CSS for debt score display, permission state badges, and explanatory text using CSS variables from tokens.css, with `@media (prefers-reduced-motion: reduce)` support
4. **Barrel export** (`src/modules/permission-debt/index.ts`)
5. **Bootstrap wiring** (`src/app/main.ts`) — import and mount `renderPermissionDebtModule()` into the `#dashboard` element after shell renders
6. **Tests** (`tests/us-023/permission-debt-module.test.ts`) — unit tests for all 5 required permissions, debt score rendering, explanatory text, unsupported permission handling, and DOM structure

### Architectural Impact Assessment

**Scope:** Small — adds a new directory (`src/modules/permission-debt/`) with four files (module, formatter, CSS, barrel), modifies one existing file (`src/app/main.ts`), and adds one new test directory (`tests/us-023/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/permissions/`, `src/scoring/`, and `src/ui/` modules. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Bootstrap flow change:** `src/app/main.ts` currently chains `renderIntro(app).then(() => renderShell(app))` with no modules mounted to `#dashboard`. The plan adds `renderPermissionDebtModule()` after shell rendering. Since `renderPermissionDebtModule()` is async (due to `checkPermissions()` returning a Promise), the mounting logic must handle the async return — either via `.then()` chaining or `await` inside the existing promise chain. This is the first module to be mounted to the dashboard, establishing the async mounting pattern for future modules. If `renderPermissionDebtModule()` throws, it should not prevent the shell from rendering — the implementation should wrap the mount in try-catch.

**Risks identified and mitigations:**

- **Async module rendering is a first for the dashboard:** The intro module is async (returns `Promise<void>`) but runs before the shell. This is the first module mounted *into* the dashboard that is async. The pattern must be established carefully — if multiple async modules mount concurrently, their order of appearance may be non-deterministic. The plan should mount the permission debt module at a predictable position (e.g., appending to `#dashboard` after awaiting). Future module mounting should follow whatever pattern is established here.

- **Clipboard permission (`clipboard-read`/`clipboard-write`) may not be queryable:** The permissions adapter (US-021) includes `clipboard-read` and `clipboard-write` in its standard list but wraps each query in try-catch, returning `'unsupported'` on failure. The format helper must handle `'unsupported'` state gracefully — the ticket notes say "Only show supported states where possible," which means unsupported permissions should either be filtered out or shown with an `'unsupported'` badge. The plan correctly identifies this and proposes filtering.

- **Color mapping inversion — granted = danger, denied = success:** The plan uses `--color-danger` for granted permissions (indicating higher debt/risk) and `--color-success` for denied/prompt permissions (indicating lower debt). This inverts the typical UX convention where green=granted. However, this inversion is correct for the "permission debt" mental model — granted permissions *increase* debt and risk. The plan's risk section correctly identifies this concern. The explanatory text should explicitly communicate this framing to avoid user confusion.

- **Acceptance criteria require exactly 5 permissions:** Notifications, Camera, Microphone, Clipboard, Location. The permissions adapter queries 11 permissions. The format helper must filter to only these 5 required permissions. Note that "Clipboard" may map to either `clipboard-read` or `clipboard-write` (or both) in the adapter — the implementation must decide which to display or whether to combine them. "Location" maps to `geolocation` in the adapter. The label mapping must be clear and correct.

- **CSS scoping — co-located vs shared:** The plan places `permission-debt.css` in the module directory (`src/modules/permission-debt/`), not in `src/styles/`. This follows the pattern established by `src/modules/intro/intro-sequence.css` for module-specific styles, which is correct — permission debt styles are not reusable across modules (unlike receipt, card, and button styles in `src/styles/`). The CSS must be imported from the module's TypeScript file to be included in the Vite bundle.

- **`PermissionDebtResult.breakdown` provides per-permission data:** The scoring module already returns a breakdown with `name`, `state`, and `weight` for each permission. The format helper can use this breakdown directly rather than re-querying or re-processing the raw `PermissionCheckResult[]`. This simplifies the data flow: `checkPermissions() → calculatePermissionDebtScore() → result.breakdown` feeds both the score display and the per-permission rows.

- **Explanatory text content:** CLAUDE.md content rules mandate "never exaggerate risk," "avoid fear language," and "use concise executive-grade wording." The explanatory text about permission debt must be factual and informative without overstating danger. Appropriate framing: explain that accumulated granted permissions increase the surface area available to web applications, and that each permission represents ongoing access that persists across visits.

- **Test mocking for `checkPermissions()`:** The async `checkPermissions()` function calls `navigator.permissions.query()` which is not available in happy-dom. Tests must mock `checkPermissions` (or the underlying `navigator.permissions`) to return predictable results. Mocking at the module level (`vi.mock('../../../src/permissions/permissions-adapter')`) is the cleanest approach, consistent with the separation of concerns — test the module's rendering logic, not the permissions API.

- **First dashboard module establishes mounting pattern:** The fingerprint receipt module (`src/modules/fingerprint/`) exists but is not yet wired into `main.ts`. This permission debt module will be the first (or among the first) to actually mount. The mounting pattern established here — how async modules append to `#dashboard`, error handling, ordering — will be followed by all subsequent modules. The plan should be explicit about DOM insertion order.

**Consistency with existing architecture:** Fully consistent. The module follows the established subdirectory pattern (`src/modules/permission-debt/` with barrel export), consistent with `src/modules/intro/` and `src/modules/fingerprint/`. The render function returns `HTMLElement` (or `Promise<HTMLElement>`), matching the pattern. The format helper separates data transformation from rendering per CLAUDE.md mandate. Co-located CSS follows the intro module precedent. The module composes existing upstream layers (`src/permissions/` → `src/scoring/` → `src/modules/`) in the expected dependency direction. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The `main.ts` modification is additive — existing intro and shell rendering are preserved.

### Consequences

- **Positive:** Delivers CLAUDE.md Required Module #4, showing users their permission posture with a quantified debt score
- **Positive:** Follows the established module pattern (subdirectory, barrel export, render function, format helper), reinforcing architectural consistency
- **Positive:** Composes two existing upstream modules (`permissions-adapter` + `permission-debt-score`) cleanly, validating the layered architecture design
- **Positive:** Establishes the async dashboard module mounting pattern that future modules will follow
- **Positive:** Reduced-motion CSS support meets CLAUDE.md accessibility requirements
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** Async mounting introduces a potential render delay — permissions queries may take varying time across browsers; mitigated by the fast return of `navigator.permissions.query()` (typically < 10ms)
- **Negative:** Color inversion (granted=danger) requires clear explanatory text to avoid user confusion — the inverted mapping is correct for the debt metaphor but counterintuitive without context
- **Trade-off:** Filtering to only 5 required permissions (from 11 queried) means the module shows a subset of available data; the remaining permissions could be surfaced in a future "expanded view" without architectural changes
