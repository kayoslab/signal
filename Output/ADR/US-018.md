## ADR-018: Add Receipt Image Export Action

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-018

### Context

The application needs a "Share Image" button on the Fingerprint Receipt so users can export the receipt as a PNG image for sharing. The `src/modules/fingerprint/fingerprint-receipt.ts` (US-012, US-013, US-015, US-017) renders a receipt with 15 signal+posture rows and a `.receipt-actions` container holding two buttons: Copy (US-016) and Re-run Audit (US-017). The `src/ui/copy-button.ts` and `src/ui/rerun-button.ts` establish the button factory pattern: pure function accepting a callback, returning `HTMLButtonElement`, with double-click guard, disabled/text/class state management, and try/catch/finally for resilience. The `src/styles/receipt.css` defines `.receipt-actions` with flexbox layout and gap spacing. The project currently has zero runtime `dependencies` in `package.json` — only `devDependencies`.

### Decision

Add a receipt image export action with:

1. **Runtime dependency** — add `html2canvas` as the project's first production dependency for client-side DOM-to-canvas rendering
2. **Share image button factory** (`src/ui/share-image-button.ts`) — `createShareImageButton(getReceiptElement: () => HTMLElement): HTMLButtonElement` following the `createCopyButton`/`createRerunButton` pattern with loading/success/error state transitions
3. **Receipt capture utility** (`src/ui/receipt-to-image.ts`) — `captureReceiptAsImage(element: HTMLElement): Promise<Blob>` wrapping `html2canvas` with white background and devicePixelRatio scaling
4. **Download trigger** — programmatic anchor click with `download='signal-receipt.png'` and object URL lifecycle management
5. **Failure handling** — try/catch with 'Export Failed' error state, reverting after 2000ms
6. **Receipt integration** — append Share Image button to `.receipt-actions` between Copy and Re-run Audit
7. **CSS states** — `.receipt-share-btn--loading`, `.receipt-share-btn--success`, `.receipt-share-btn--error` in `src/styles/receipt.css`
8. **Unit tests** (`tests/us-018/`) — button creation, click flow with mocked html2canvas, state transitions, graceful failure, receipt-to-image blob return

### Architectural Impact Assessment

**Scope:** Small — adds two new source files to `src/ui/` (fourth and fifth components after `receipt.ts`, `rerun-button.ts`, `copy-button.ts`), modifies two existing files (`src/modules/fingerprint/fingerprint-receipt.ts`, `src/styles/receipt.css`), adds one new test directory (`tests/us-018/`), and adds `html2canvas` as the project's first runtime dependency.

**New dependencies:** `html2canvas` — production dependency (~40KB gzipped). This is a client-side-only library that renders DOM elements to canvas without any external network calls, analytics, or tracking. It performs all rendering locally in the browser using canvas APIs. **Architecture policy compliance verified:** `html2canvas` does not match any `BANNED_ANALYTICS_PATTERNS`, `BANNED_TRACKING_PATTERNS`, or `BANNED_BACKEND_PATTERNS` in `src/policies/architecture-policy.ts`. It makes no external HTTP requests and processes no user data beyond the DOM element passed to it. It complies with all CLAUDE.md Non-Negotiable Trust Rules (no cookies, no analytics, no tracking, no backend, no external data exfiltration, all processing local). However, this is the project's **first runtime dependency** — `package.json` currently has zero `dependencies`, only `devDependencies`. This is a notable precedent.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, `npm run test`, and `npm run policy:check` scripts cover all new and modified files without modification. The architecture policy validator checks `dependencies` against banned patterns — `html2canvas` will pass validation. The runtime bundle size will increase by ~40KB gzipped.

**Risks identified and mitigations:**

- **First runtime dependency sets a precedent:** The project has maintained zero production dependencies, keeping the bundle purely first-party code. Adding `html2canvas` opens the door to future runtime dependency additions. This is justified — the alternative (manual Canvas API rendering of the entire receipt layout) would require reimplementing CSS layout, typography, borders, and spacing in canvas draw calls, which is fragile, hard to maintain, and would likely exceed the complexity of the library itself. The ticket notes explicitly say "use client-side rendering library." The ~40KB gzipped cost is acceptable for a static site. **Mitigation:** Consider lazy-loading via dynamic `import()` so `html2canvas` is only fetched when the user clicks Share Image, avoiding bundle size impact for users who never use the feature. The plan identifies this optimization but does not require it for the initial implementation.

- **html2canvas rendering fidelity:** html2canvas re-renders DOM using canvas — some CSS features (custom fonts, complex shadows, CSS variables, pseudo-elements) may not render perfectly. The receipt uses monospace fonts (`font-family: monospace` in `receipt.css`), simple flexbox layout, dotted borders, and design token colors. Monospace system fonts and simple layouts are well-supported by html2canvas. However, CSS custom properties (`var(--color-*)`) are resolved at render time by the browser before html2canvas reads computed styles, so token-based colors will render correctly. The dotted leader pattern in `.receipt-row` (using `border-bottom: 1px dotted`) should render reliably. **Visual verification is required** during implementation.

- **`canvas.toBlob()` returning null:** The `toBlob()` method can return `null` in certain browser states (e.g., tainted canvas, insufficient memory). The plan's error handling in step 5 must treat `null` blob as a failure, showing 'Export Failed'. This is explicitly addressed in the plan.

- **Mobile download behavior (iOS Safari):** Programmatic downloads via temporary anchor click (`<a download="..." href="blob:...">`) do not work reliably on iOS Safari — the `download` attribute is not honored, and the browser may open the image in a new tab instead. The plan's risk section correctly identifies this. **Mitigation:** Accept the degraded behavior on iOS (image opens in new tab where users can long-press to save) or detect iOS and fall back to `window.open(objectURL)`. This is a UX concern, not an architectural risk — the feature remains functional.

- **Object URL lifecycle:** The plan creates an object URL from the blob, attaches it to a temporary anchor, clicks it, then revokes the URL. If revocation happens synchronously after the click, some browsers may not complete the download. The implementation should use `setTimeout(() => URL.revokeObjectURL(url), 1000)` or similar to allow the download to initiate before cleanup. Standard pattern for blob downloads.

- **Timer-based state revert (2s setTimeout) and component lifecycle:** Same pattern as `copy-button.ts` and `rerun-button.ts` — if the receipt is removed from DOM before timeout fires, the callback executes on a detached element harmlessly. Acceptable for the static SPA architecture.

- **Existing test compatibility:** The Share Image button is added to `.receipt-actions` as a sibling of the Copy and Rerun buttons. US-013 tests assert `expect(rows.length).toBe(15)` on `.receipt-row` elements inside `.receipt-rows` — the button container is separate and unaffected. US-016 and US-017 tests verify individual button behavior without counting siblings. **No test regressions expected.** However, if any integration test asserts on `.receipt-actions` child count (none currently do), it would need updating from 2 to 3 children.

- **Button order in `.receipt-actions`:** The plan specifies Copy | Share Image | Re-run Audit. Currently the order is Copy (prepended) | Re-run Audit (appended). The Share Image button should be inserted between them. The implementation can use `insertBefore(shareBtn, rerunBtn)` or append in the correct order. This is a minor DOM ordering concern.

- **Three-button layout on mobile:** `.receipt-actions` uses `display: flex; justify-content: flex-end; gap: var(--spacing-sm)`. Three buttons at mobile widths (320px) may overflow or look cramped. The plan correctly identifies adding `flex-wrap: wrap` to `.receipt-actions` as a potential fix. Each button has compact text ('Copy', 'Share Image', 'Re-run Audit') — at `--font-size-sm` these should fit at 320px without wrapping, but visual verification is needed.

- **html2canvas import in tests:** Vitest runs in Node/jsdom where `HTMLCanvasElement` and `getContext('2d')` may not be fully functional. Tests must mock `html2canvas` at the module level (`vi.mock('html2canvas')`) to return a stub canvas with a mock `toBlob` method. This is the standard approach — test the integration logic (button states, download trigger), not the html2canvas rendering itself.

**Consistency with existing architecture:** Fully consistent with established patterns. The share image button factory follows the exact pattern from `createCopyButton` and `createRerunButton`: pure function accepting typed input, returning `HTMLButtonElement`, with state management via class toggles and disabled property, double-click guard, try/catch/finally for resilience, and 2000ms revert timer. CSS additions to `src/styles/receipt.css` follow the existing BEM modifier class pattern (`--loading`, `--success`, `--error`). The `receipt-to-image.ts` utility separates the capture concern from the button concern, consistent with CLAUDE.md's separation of data/rendering logic. The receipt integration follows the existing button wiring pattern in `fingerprint-receipt.ts`. Tests follow the `tests/us-{number}/` convention. All processing remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The `html2canvas` dependency addition is the only change affecting the project's dependency profile.

### Consequences

- **Positive:** Delivers image export functionality for the fingerprint receipt, directly fulfilling all four US-018 acceptance criteria (Share Image button visible, receipt exported as PNG, downloaded image readable, failure state handled gracefully)
- **Positive:** Follows the established button factory pattern from `createCopyButton`/`createRerunButton`, maintaining architectural consistency in `src/ui/`
- **Positive:** `html2canvas` is client-side only with no external network calls, fully compliant with CLAUDE.md Non-Negotiable Trust Rules and architecture policy
- **Positive:** Separation of capture logic (`receipt-to-image.ts`) from button logic (`share-image-button.ts`) keeps concerns isolated and independently testable
- **Positive:** All data remains client-side; no trust rule violations — the PNG is generated locally and downloaded directly
- **Negative:** Introduces the project's first runtime dependency (`html2canvas`, ~40KB gzipped), increasing bundle size. This is the most significant architectural consequence — the project moves from zero runtime dependencies to one. Mitigated by the library's local-only nature and the alternative's impracticality (manual Canvas API rendering)
- **Negative:** html2canvas rendering fidelity is not guaranteed for all CSS features — visual verification required during implementation. The receipt's simple layout (monospace text, flexbox rows, dotted borders) is well within html2canvas's capabilities
- **Negative:** iOS Safari does not reliably support programmatic `<a download>` — users may see the image in a new tab instead of an automatic download. Functionally equivalent (user can still save the image) but not the ideal UX
- **Trade-off:** Lazy-loading html2canvas via dynamic `import()` would eliminate the bundle size impact for users who never click Share Image, but adds async loading complexity to the click handler. Recommended as a follow-up optimization, not a blocker for initial implementation
