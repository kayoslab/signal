## ADR-035: Validate Production Static Build

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-035

### Context

The application has accumulated 7 dashboard modules, 3 signal collectors, 2 scoring modules, 5 inference rules, 4 threat rules, 7 hardening rules, accessibility improvements, and reduced-motion support across 30+ source files and 27 test directories. US-035 is the release validation ticket — it verifies that the full codebase produces a clean production build suitable for static hosting deployment. The build pipeline (`npm run build`) already includes an architecture policy pre-check gate (`npm run policy:check && vite build`) established in ADR-003. All upstream dependencies (US-032, US-033, US-034) are complete.

### Decision

Validate the production static build through a sequential verification pipeline:

1. **Install dependencies** — `npm install` to ensure all CLI tools are available
2. **Run lint** — `npm run lint` (ESLint on `src/`) and fix any errors
3. **Run typecheck** — `npm run typecheck` (tsc on both `src/` and `tests/`) and fix any errors
4. **Run tests** — `npm run test` and ensure all existing unit tests pass
5. **Run build** — `npm run build` (policy:check + vite build) producing `dist/` with index.html and bundled assets
6. **Run preview** — `npm run preview` for manual verification of all 7 modules rendering without console errors
7. **Write validation tests** — `tests/us-035/production-build.test.ts` asserting: (a) `npm run build` exits 0, (b) `dist/index.html` exists, (c) `dist/assets/` contains JS and CSS bundles, (d) no raw `/src/` TypeScript references remain in built HTML, (e) built index.html references hashed assets
8. **Final end-to-end verification** — lint, typecheck, test, build pass in sequence
9. **Mark ticket passes=true and commit**

### Architectural Impact Assessment

**Scope:** Very small — adds one new test directory (`tests/us-035/`) with one test file. No source code changes unless lint/typecheck errors require fixes. No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The validation tests use `node:child_process` (for `execSync` build invocation), `node:fs`, and `node:path` — all Node.js built-in modules already used in `tests/us-003/static-deploy.test.ts`.

**Build pipeline change:** None. The existing `npm run build` pipeline (policy:check + vite build) is validated, not modified.

**Risks identified and mitigations:**
- **Upstream lint/typecheck regressions from US-032/033/034 could block the build** — mitigated by sequential pipeline catching errors early and all upstream tickets already passing their deployment gates
- **html2canvas is listed as devDependency but imported at runtime** — Vite bundles all imports regardless of dependency classification, so this is a non-issue per ADR-018
- **vite preview requires manual browser verification for console errors and module rendering** — cannot be fully automated without Playwright (deferred per ADR-034)
- **Build test creates dist/ as side effect which could affect test isolation** — mitigated by dist/ being in .gitignore and builds being idempotent
- **New test in tests/us-035/ could be accidentally excluded by test script patterns** — verified safe: exclusion patterns only match us-001 and us-002

**Consistency with existing architecture:** Fully consistent. The validation test follows the exact pattern established in `tests/us-003/static-deploy.test.ts`: shell out to build via `execSync`, assert on `dist/` structure, verify no policy violations in output. Tests land in `tests/us-035/` following the per-ticket convention. No runtime code changes. No new patterns introduced. No dependencies added. All verification is local per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Provides automated regression protection for the production build — future changes that break the build will be caught by the exit-code assertion
- **Positive:** Verifies that Vite correctly replaces dev entry points with hashed production assets, catching misconfigured build settings
- **Positive:** No raw `/src/` references assertion ensures the production bundle is self-contained and suitable for static hosting without a dev server
- **Positive:** Sequential pipeline (lint → typecheck → test → build) catches issues at the earliest possible stage
- **Positive:** Zero new dependencies — validation uses existing Node.js built-in modules and established test patterns
- **Negative:** `npm run build` side effect in tests creates `dist/` directory — mitigated by `.gitignore` and the build being idempotent
- **Negative:** Manual `vite preview` step cannot be automated without Playwright — deferred to a future ticket per ADR-034
- **Trade-off:** The validation test overlaps slightly with `tests/us-003/static-deploy.test.ts` (dist/ structure verification) — acceptable because US-035 adds hashed-asset and no-raw-src assertions that US-003 does not cover, serving as an end-to-end build verification checkpoint
