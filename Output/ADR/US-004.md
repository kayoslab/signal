## ADR-004: Responsive Application Shell Layout

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-004

### Context

The application needs a responsive layout shell (header, main content area, footer) that provides a three-column grid on desktop (≥1200px) and stacks vertically on mobile. The `src/layout/` directory exists but is empty (`.gitkeep` only). The current `src/app/main.ts` renders a placeholder string into the `#app` div. No DOM testing environment is configured — Vitest runs in the default Node environment.

### Decision

Implement a responsive shell layout with the following changes:

1. **Design tokens** (`src/styles/tokens.css`) — CSS custom properties for colors, spacing, typography, and responsive breakpoints aligned with CLAUDE.md visual direction
2. **Layout module** (`src/layout/shell.ts`) — pure function `createShell()` returning a DocumentFragment with semantic `<header>`, `<main>`, and `<footer>` elements
3. **Scoped CSS** (`src/layout/shell.css`) — mobile-first single-column grid, with `@media (min-width: 1200px)` breakpoint switching to three-column desktop layout
4. **Global styles update** — import tokens.css, add `overflow-x: hidden` and `min-height: 100vh` to body rule in `src/styles/global.css`
5. **Entry point integration** — replace the placeholder text in `src/app/main.ts` with shell fragment insertion
6. **Test environment** — install `happy-dom` (devDependency) and add `environment: 'happy-dom'` and `css: true` to `vitest.config.ts`
7. **Tests** (`tests/us-004/shell.test.ts`) — DOM structure assertions via happy-dom, plus static CSS content parsing to verify responsive rules and overflow placement

### Architectural Impact Assessment

**Scope:** Small-to-medium — adds one devDependency (`happy-dom`), two new source files in an existing directory (`src/layout/shell.ts`, `src/layout/shell.css`), one new tokens file (`src/styles/tokens.css`), modifies three existing files (`vitest.config.ts`, `src/styles/global.css`, `src/app/main.ts`), and adds one test directory (`tests/us-004/`).

**New dependencies:** `happy-dom` (devDependency only). This is a standard lightweight DOM implementation for testing that does not affect the runtime bundle. No new runtime dependencies are introduced.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new files without modification. The `build` script (with policy pre-check gate from US-003) is unaffected.

**Configuration change:** `vitest.config.ts` gains `environment: 'happy-dom'` and `css: true`. This affects all test files globally and is the highest-impact change.

**Risks identified and mitigations:**
- **Global happy-dom environment affecting existing tests:** Existing US-003 tests (`architecture-policy.test.ts`, `static-deploy.test.ts`) import `node:fs`, `node:path`, and `node:child_process`. happy-dom adds browser globals (`document`, `window`) on top of the Node environment — it does not remove Node APIs. These tests should continue to work. US-006 and US-007 tests already mock browser globals (`navigator`, `Intl`, `window`) and will benefit from happy-dom providing base browser APIs. If any test breaks, per-file `// @vitest-environment node` overrides can restore the original environment for that specific file.
- **CSS static analysis tests vs runtime rendering:** happy-dom does not compute CSS Grid layout or evaluate media queries at runtime. The plan correctly addresses this by parsing CSS file content as text rather than testing computed styles. This is the right approach.
- **`overflow-x: hidden` as horizontal scroll prevention:** While effective as a safety net, the plan's use of `fr` units in the grid inherently prevents content overflow. The `overflow-x: hidden` in global.css is defense-in-depth, not the primary mechanism. Should be paired with `box-sizing: border-box` (already present in global.css reset) to prevent sizing surprises.
- **Intermediate viewport gap (768px–1200px):** The plan mentions a `--breakpoint-md: 768px` token but does not use it for any layout rules. US-004 acceptance criteria only require desktop three-column (≥1200px) and mobile vertical stacking — this is acceptable. The token exists for future use.
- **tokens.css not in the ADR but in the plan:** The plan adds `src/styles/tokens.css` for design tokens. This is a reasonable addition that establishes a centralized token system for future tickets. It follows CLAUDE.md's `src/styles/` directory convention.

**Consistency with existing architecture:** Fully consistent. Files land in `src/layout/` and `src/styles/` (designated directories per CLAUDE.md). CSS follows the project's scoped approach with import from the TypeScript module. The shell uses vanilla TypeScript with `document.createElement` — no framework dependencies. The mobile-first responsive pattern aligns with the project's lightweight, static-hosting approach. Tests follow the `tests/us-{number}/` convention established by prior tickets.

**No breaking changes** to existing code or build pipeline. The vitest environment change is the only modification touching existing test infrastructure, and it is additive (adds DOM globals) rather than destructive (does not remove Node APIs).

### Consequences

- **Positive:** Establishes the application's structural foundation — all future module tickets (Fingerprint Receipt, OSINT Cards, Permission Debt, etc.) will mount into the shell's `<main>` content area
- **Positive:** Design tokens file creates a single source of truth for visual constants, enabling consistent styling across future modules
- **Positive:** happy-dom environment enables DOM-level testing for all future UI tickets without per-file configuration
- **Positive:** CSS Grid with `fr` units provides flexible, responsive layout with minimal CSS
- **Positive:** Zero runtime dependencies; layout is pure vanilla TypeScript + CSS
- **Negative:** Global happy-dom environment change requires verification that all existing tests (US-001 through US-003, US-006, US-007) still pass — minor migration risk mitigated by per-file environment override escape hatch
- **Trade-off:** Static CSS text parsing tests verify rule presence but not visual correctness; Playwright E2E tests (required by CLAUDE.md before release) will provide the visual verification layer
