## ADR-010: Animated Intro Sequence Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-010

### Context

The application needs an animated opening sequence that displays a signal count and privacy statement before the main dashboard appears (CLAUDE.md Required Module #1). The `src/modules/` directory exists but is empty (`.gitkeep` only). The bootstrap in `src/app/main.ts` currently calls `renderShell(app)` synchronously. The `src/signals/snapshot.ts` (US-009) provides `collectSnapshot()` returning a frozen `SignalSnapshot` with `locale`, `device`, and `rendering` sub-objects. Design tokens (US-005) and the responsive shell (US-004) are established.

### Decision

Add an animated intro sequence module with:

1. **Signal counter** (`src/modules/intro/count-signals.ts`) â€” pure function `countSignalFields(snapshot: SignalSnapshot): number` that recursively counts meaningful (non-sentinel) fields from the snapshot object, separating data logic from rendering per CLAUDE.md rules.
2. **Intro component** (`src/modules/intro/intro-sequence.ts`) â€” `renderIntro(root: HTMLElement): Promise<void>` that collects the snapshot, derives signal count, renders a fullscreen overlay with intro text, signal count, and privacy statement, animates over ~2s, then auto-removes the overlay.
3. **Intro styles** (`src/modules/intro/intro-sequence.css`) â€” scoped CSS using existing design tokens, keyframe animations for text reveal, with `@media (prefers-reduced-motion: reduce)` block replacing animations with instant opacity transitions.
4. **Bootstrap change** (`src/app/main.ts`) â€” change from synchronous `renderShell(app)` to async flow: `await renderIntro(app)` then `renderShell(app)`.
5. **Tests** (`tests/us-010/intro-sequence.test.ts`, `tests/us-010/count-signals.test.ts`) â€” unit tests for signal counting and DOM structure/timing tests for the intro component.

### Architectural Impact Assessment

**Scope:** Small-to-medium â€” adds a new directory (`src/modules/intro/`) with three files, modifies one existing file (`src/app/main.ts`), and adds one new test directory (`tests/us-010/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module uses standard DOM APIs, CSS animations, and imports from the existing `src/signals/` barrel export. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Bootstrap flow change:** This is the most significant architectural change. `src/app/main.ts` currently executes synchronously: `renderShell(app)`. The plan changes this to an async sequence: `renderIntro(app)` (returns `Promise<void>`) followed by `renderShell(app)`. This means:
- The entry point must handle the async nature (top-level await or `.then()` chain)
- The shell is no longer visible on initial render â€” the intro overlay gates it
- If `renderIntro` throws, the shell never renders unless error handling is added
- The `collectSnapshot()` call moves from a future dashboard module into the intro sequence, establishing the intro as the first consumer of the signal pipeline

**Risks identified and mitigations:**
- **Async bootstrap error handling:** If `collectSnapshot()` or the intro animation fails, the dashboard would never appear. The implementation should wrap the intro in try-catch with a fallback that still renders the shell. This is an implementation concern the plan does not explicitly address but is straightforward to handle.
- **`collectSnapshot()` called inside intro creates a coupling:** The intro component imports and calls `collectSnapshot()` to derive the signal count. This means the intro depends on all three signal collectors (locale, device, rendering). If any collector has a bug that throws (despite their try-catch design), it could block the intro. The frozen snapshot's try-catch-per-collector design mitigates this â€” each collector handles its own errors. Risk is low.
- **Animation timing budget:** CSS animations must sum to â‰¤2s (acceptance criterion). With 3 text lines at ~400ms each + fade-out, the budget is tight. CSS `animation-duration` values are deterministic, so this is a design constraint, not an architectural risk. The plan correctly identifies this.
- **jsdom/happy-dom animation limitations:** The test environment does not support CSS animations or `matchMedia`. Tests must verify DOM structure and class application, not actual animation behavior. The plan acknowledges this and uses `vi.useFakeTimers()` for timing tests. Note: `vitest.config.ts` currently has no `environment` setting â€” tests may need per-file `// @vitest-environment happy-dom` directives or the global config needs updating (ADR-004 planned this but the config only has `include`).
- **`prefers-reduced-motion` requires both CSS and JS handling:** CSS removes animations, but if the JS uses `setTimeout`/`requestAnimationFrame` for sequencing, reduced-motion users still wait through timed delays. The implementation should check `matchMedia('(prefers-reduced-motion: reduce)')` in JS to skip delays entirely, not just rely on CSS. The plan's risk section correctly identifies this dual requirement.
- **Signal count determinism:** `countSignalFields` must define what constitutes a "meaningful" field â€” it needs to skip sentinel values (`'unknown'`, `'unavailable'`) and metadata fields (`collectedAt`, `version`). The `SignalSnapshot` contains nested objects (`locale`, `device`, `rendering`) with mixed types (strings, numbers, booleans, arrays, objects like `storageSupport`). The recursive counting logic must handle: string sentinels, boolean values (always meaningful), number values (always meaningful), arrays (`languages` â€” count as 1 or by length?), and nested objects (`storageSupport` â€” recurse into). These are implementation decisions that affect the displayed count but not the architecture.
- **First module in `src/modules/` sets the pattern:** This is the first module to populate the `src/modules/` directory. The plan's structure (`src/modules/intro/count-signals.ts`, `src/modules/intro/intro-sequence.ts`, `src/modules/intro/intro-sequence.css`) establishes a convention of per-module subdirectories with co-located CSS. This differs from the `src/styles/` shared CSS pattern used by US-005 (card.css, button.css) and the `src/layout/` co-located pattern (shell.ts + shell.css). Co-locating the intro CSS with its component is the correct choice since the intro styles are module-specific, not shared. Future module authors should follow this convention for module-specific styles.

**Consistency with existing architecture:** Fully consistent. Files land in `src/modules/` per CLAUDE.md conventions (designated directory for dashboard panels). The pure function for signal counting follows the separation-of-concerns principle (data logic separate from rendering). The component uses the established pattern of pure functions with `document.createElement` (consistent with `shell.ts` and `src/ui/receipt.ts`). CSS uses existing design tokens from `src/styles/tokens.css`. Tests land in `tests/us-010/` following the per-ticket convention. No runtime dependencies added. All data remains client-side per trust rules. The privacy statement content aligns with CLAUDE.md Non-Negotiable Trust Rules.

**No breaking changes** to existing tests or build pipeline. The `src/app/main.ts` modification changes application behavior (intro gates the shell) but does not break any existing module â€” the shell still renders, just after the intro completes.

### Consequences

- **Positive:** Establishes the first dashboard module and a reusable pattern (per-module subdirectory with co-located styles, pure functions, typed interfaces) for future modules (Fingerprint Receipt, OSINT Cards, Permission Debt, etc.)
- **Positive:** The signal count provides immediate user value by demonstrating the scope of passive browser exposure before the detailed dashboard appears
- **Positive:** Privacy statement in the intro reinforces the trust contract before any analysis results are shown
- **Positive:** Reduced-motion support via CSS media query and potential JS `matchMedia` check meets CLAUDE.md accessibility requirements
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** Async bootstrap introduces a potential failure mode where the shell never renders if the intro throws â€” mitigated by try-catch in the implementation
- **Negative:** The intro component couples to the full signal pipeline (all three collectors) â€” acceptable since this is the module's core purpose and the collectors are designed to be failure-resilient
- **Trade-off:** Co-located CSS in `src/modules/intro/` rather than `src/styles/` is appropriate for module-specific styles but diverges from the shared component pattern â€” this is intentional and correct since intro styles are not reusable
