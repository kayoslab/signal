## ADR-019: Zero Click OSINT Card Component

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-019

### Context

The application needs a reusable intelligence card component for displaying passive OSINT findings (CLAUDE.md Required Module #3). Each card must show a title, value, source, confidence level, and "why it matters" explanation. The `src/ui/` directory contains two components (`receipt.ts`, `rerun-button.ts`) establishing the pattern: pure functions using `document.createElement`, returning `HTMLElement`, with exported TypeScript interfaces. The `src/styles/` directory has a complete design token system (`tokens.css`, US-004/US-005) including semantic color tokens (`--color-success`, `--color-warning`, `--color-danger`), a base `.card` class (`card.css`, US-005), and shared component CSS imported via `global.css`.

### Decision

Add a reusable OSINT card component with:

1. **Data interface and component** (`src/ui/osint-card.ts`) — `OsintCardData` type with `title`, `value`, `source`, `confidence` (`'high' | 'medium' | 'low'`), and `whyItMatters` fields. Pure function `createOsintCard(data: OsintCardData): HTMLElement` returning a semantic card element with BEM-style classes (`.osint-card`, `.osint-card-title`, `.osint-card-value`, `.osint-card-source`, `.osint-card-confidence`, `.osint-card-why`) and a `data-confidence` attribute for CSS-driven color variants. Convenience function `createOsintCardList(cards: OsintCardData[]): HTMLElement` wrapping multiple cards in a `.osint-card-list` responsive grid container.
2. **Scoped styles** (`src/styles/osint-card.css`) — extends `.card` base pattern with confidence badge color mapping via `[data-confidence]` attribute selectors using existing semantic tokens (`--color-success` for high, `--color-warning` for medium, `--color-danger` for low). Responsive grid layout for `.osint-card-list` (1 column mobile, 2 columns at `--breakpoint-md`). Typography follows financial-report precision aesthetic using existing font tokens. Includes `prefers-reduced-motion` support.
3. **Global CSS import** — `@import './osint-card.css'` added to `src/styles/global.css` after the existing `receipt.css` import.
4. **Tests** (`tests/us-019/osint-card.test.ts`) — unit tests covering all five acceptance criteria independently, confidence level attribute mapping for all three levels, `createOsintCardList` child count verification, and CSS file token usage verification.

### Architectural Impact Assessment

**Scope:** Very small — adds one TypeScript file to `src/ui/` (third component after `receipt.ts` and `rerun-button.ts`), one CSS file to `src/styles/`, modifies one line in `src/styles/global.css`, and adds one test directory (`tests/us-019/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The component uses standard DOM APIs available in Vitest's happy-dom environment (configured in US-004). No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**
- **Confidence badge color mapping must use existing semantic tokens:** Verified — `--color-success` (#22c55e), `--color-warning` (#eab308), and `--color-danger` (#ef4444) all exist in `src/styles/tokens.css`. No token creation needed. Risk eliminated.
- **CSS class naming collision with `.card` base styles:** The plan uses `.osint-card` prefix throughout (BEM-style), which will not collide with the existing `.card` class in `card.css`. The component composes with `.card` (adding both classes to the root element) to inherit base card styles (background, border, radius, shadow, padding), then extends via `.osint-card` specificity. This is the correct CSS composition pattern, consistent with how `.receipt` relates to `.card`.
- **The existing `src/styles/card.css` base `.card` class is minimal:** Verified — `card.css` provides surface background, border, border-radius, box-shadow, and padding using design tokens. This is sufficient foundation for the OSINT card. The plan extends rather than modifies the base `.card` class, so no risk to the fingerprint receipt module or other `.card` consumers.
- **WCAG contrast for confidence badge colors:** The semantic color tokens (`--color-success`, `--color-warning`, `--color-danger`) are used as badge background colors against the card surface. `--color-warning` (#eab308, yellow) may have insufficient contrast for white text — the implementation should use dark text on warning badges or adjust the badge design to use colored borders/outlines rather than colored backgrounds with text. This is an implementation concern the developer should verify.
- **`data-confidence` attribute approach:** Using a data attribute for CSS-driven color variants is a clean pattern that avoids conditional class generation in JavaScript. The CSS uses `[data-confidence="high"]`, `[data-confidence="medium"]`, `[data-confidence="low"]` attribute selectors. This is more maintainable than adding/removing modifier classes.
- **Reusability across multiple modules:** The `OsintCardData` interface maps directly to the CLAUDE.md requirement for OSINT cards (finding, source, confidence, why it matters). The component is purely presentational with no signal-collection logic mixed in, per CLAUDE.md coding rules about separating data collection from rendering.

**Consistency with existing architecture:** Fully consistent. The component follows the exact pattern established by `src/ui/receipt.ts`: exported TypeScript interface, pure function accepting typed input, returning `HTMLElement`, using `document.createElement` with class assignment. CSS follows the shared component pattern from US-005 (`card.css`, `button.css`, `receipt.css`) — placed in `src/styles/` and imported via `global.css`. Tests follow the `tests/us-{number}/` convention with DOM structure assertions plus CSS content verification. No runtime dependencies added. All rendering is client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Adds the third `src/ui/` component, reinforcing the established pure-function component pattern and providing the card primitive needed by the Zero Click OSINT module (CLAUDE.md Required Module #3)
- **Positive:** The `OsintCardData` interface provides a clear data contract for upstream modules that generate passive findings
- **Positive:** Confidence badge with semantic color tokens provides immediate visual risk communication aligned with CLAUDE.md's visual direction (financial-report precision, security operations center clarity)
- **Positive:** `createOsintCardList` convenience function provides responsive grid layout for multiple cards, reducing boilerplate in consuming modules
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** `data-confidence` attribute approach is accessible to CSS and to assistive technologies via attribute selectors
- **Negative:** None significant — this is a minimal, additive change with no architectural risk
- **Trade-off:** Global CSS import means OSINT card styles are loaded even if the component is not used on a given view; negligible impact given the small CSS footprint and static single-page architecture
