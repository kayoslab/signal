## ADR-033: Add Reduced Motion Behavior

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-033

### Context

The application has partial `prefers-reduced-motion` support across five CSS files and one TypeScript file, but coverage is inconsistent. The intro sequence CSS (`src/modules/intro/intro-sequence.css`) reduces transitions from 300‚Äì400ms to 100ms rather than eliminating them. The intro TypeScript (`src/modules/intro/intro-sequence.ts`) checks `matchMedia` but still imposes an 800ms hold delay and a 100ms fade-out in the reduced-motion path. `src/styles/button.css` has `transition` properties but no `prefers-reduced-motion` override, relying on the `shell.css` global catch-all which only covers elements inside `.shell`. Additionally, `.intro-line` elements start with `opacity: 0` by default, creating a brief invisible-content frame before JS adds the `--visible` class ‚Äî content could be hidden if JS is slow or fails.

### Decision

Harden reduced-motion support across all CSS and JS paths:

1. **Intro CSS** ‚Äî change `transition: opacity 100ms ease-out` to `transition: none` for both `.intro-line` and `.intro-overlay` under `prefers-reduced-motion`, and add `.intro-line { opacity: 1; transform: none; }` to ensure content is visible by default without JS class addition
2. **Intro JS** ‚Äî reduce the 800ms hold delay to ~50ms or minimal, and skip the fade-out overlay removal when `reducedMotion` is true, so no perceptible animation occurs
3. **Button CSS** ‚Äî add explicit `@media (prefers-reduced-motion: reduce)` override setting `transition: none` for `.btn`
4. **Tests** ‚Äî new test file (`tests/us-033/reduced-motion.test.ts`) verifying CSS `transition: none` under reduced motion, JS completion timing, static scan of all CSS files for reduced-motion coverage, and content visibility defaults
5. **Verify** the global `shell.css` catch-all still functions as a safety net

### Architectural Impact Assessment

**Scope:** Very small ‚Äî modifies three existing files (`src/modules/intro/intro-sequence.css`, `src/modules/intro/intro-sequence.ts`, `src/styles/button.css`), adds one new test directory (`tests/us-033/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. All changes use existing CSS media queries, standard `window.matchMedia` API, and Vitest test infrastructure already configured in the project.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all modified and new files without modification.

**Risks identified and mitigations:**

- **Over-aggressive transition removal could make interactive feedback feel broken:** The plan correctly identifies this risk. Button hover/focus visual changes still occur instantly via CSS property changes ‚Äî `transition: none` means the property change happens in zero time, not that the visual change disappears. The `outline` on `.btn:focus-visible` is not animated (it appears/disappears with focus state), so accessibility focus indicators are unaffected. Low risk.

- **Global shell.css catch-all uses `!important` which could conflict with component-level rules:** The global rule sets `transition-duration: 0.01ms !important`. Component-level `transition: none` is equivalent in effect (no perceptible transition). If a component rule sets `transition: none` and the global sets `transition-duration: 0.01ms !important`, the result is still effectively instant. No functional conflict ‚Äî both approaches produce the same user-visible behavior. The component-level rules provide explicit documentation of intent and cover elements that may render outside `.shell` (like the intro overlay at `z-index: 9999` which is appended to `root`, not inside `.shell`).

- **Intro overlay rendered outside `.shell` is NOT covered by the global catch-all:** This is verified ‚Äî `renderIntro(root)` appends the overlay to `root` (the `#app` element), not inside `.shell`. The `.shell` is rendered *after* the intro completes. Therefore, the intro overlay's `.intro-line` and `.intro-overlay` elements are NOT descendants of `.shell` and the global catch-all rule does not apply to them. The explicit intro-level CSS and JS reduced-motion handling are the only protections for the intro sequence. This makes hardening the intro CSS (plan steps 2‚Äì4) essential, not redundant.

- **Reducing the 800ms hold delay to ~50ms could prevent users from reading the privacy statement:** The plan's risk section correctly identifies this concern. With reduced motion enabled, all three intro lines appear immediately (opacity: 1 from CSS, plus JS adds `--visible` class). The ~50ms hold is just to ensure DOM painting completes before overlay removal. Users who prefer reduced motion are explicitly choosing faster, less animated interfaces ‚Äî a near-instant intro is the expected behavior. The privacy statement is also displayed permanently in the footer (US-031), so no content is lost.

- **Existing US-010 tests assert intro timing:** `tests/us-010/intro-sequence.test.ts` has a test "completes within 2 seconds even with reduced motion" that mocks `matchMedia` to return `matches: true`. The current implementation takes 800ms + 100ms = 900ms in reduced-motion mode. Reducing the hold to ~50ms means the intro completes in ~50ms, which still passes the "within 2 seconds" assertion. **No test regression.** If a new US-033 test asserts completion under 200ms, that is additive and does not conflict.

- **CSS-only fix for content visibility (AC3):** Adding `.intro-line { opacity: 1; transform: none; }` under the reduced-motion media query ensures content is visible even before JS executes. This is defense-in-depth ‚Äî if JS is delayed or fails, users with reduced motion still see content rather than a blank screen. The `--visible` class addition in JS becomes a no-op since the properties are already at their target values. No conflict.

- **Static CSS scan test may be brittle:** The plan proposes scanning all CSS files to verify that any file with `transition` or `animation` also contains `prefers-reduced-motion`. After this plan's changes, all CSS files with transitions/animations have explicit reduced-motion overrides: `intro-sequence.css`, `button.css`, `receipt.css`, `osint-card.css`, `permission-debt.css`, plus the `shell.css` global catch-all. The only file that currently lacks an explicit override is `button.css`, which this plan fixes. The static scan is a reasonable regression-prevention measure. Note: `tokens.css` defines `--transition-fast` and `--transition-base` tokens but these are definitions, not applied transitions ‚Äî the scan should check for `transition:` or `animation:` property declarations, not token definitions.

- **No changes to `tokens.css`:** The plan mentions considering reduced-motion token overrides in `tokens.css` but does not require them. This is correct ‚Äî overriding transition tokens under a media query in `tokens.css` would create a global behavior change that could have unintended effects on any future component using those tokens. Component-level overrides are more explicit and predictable.

- **Button hover still provides visual feedback:** With `transition: none`, the `.btn:hover { opacity: 0.9 }` rule still applies ‚Äî the opacity change happens instantly rather than smoothly. This is the correct behavior for reduced-motion preferences: property changes are immediate, not animated.

**Consistency with existing architecture:** Fully consistent. The changes strengthen existing patterns rather than introducing new ones. The intro CSS and JS already have `prefers-reduced-motion` support; this plan hardens it from "reduced" to "eliminated." The `button.css` override follows the same pattern used in `osint-card.css`, `receipt.css`, and `permission-debt.css`. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All changes are client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The US-010 intro timing test (900ms ‚Üí ~50ms under reduced motion) still passes its "within 2 seconds" assertion. The US-013 row count test is unaffected. No existing test asserts on specific transition durations in the intro CSS.

### Consequences

- **Positive:** Eliminates all perceptible animation for users who prefer reduced motion, fully satisfying CLAUDE.md accessibility requirements
- **Positive:** CSS-only content visibility fix (AC3) ensures intro content is never hidden even if JS is delayed, providing progressive enhancement
- **Positive:** Explicit `button.css` reduced-motion override removes reliance on the `.shell` global catch-all for buttons that may render outside the shell
- **Positive:** Static CSS scan test prevents future regressions ‚Äî any new CSS file with transitions must include reduced-motion support
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All changes are client-side; no trust rule violations
- **Negative:** None significant ‚Äî this is a minimal, targeted hardening of existing accessibility support with no architectural risk
- **Trade-off:** Near-instant intro under reduced motion means users see the privacy statement very briefly ‚Äî mitigated by the persistent privacy footer (US-031) displaying the same trust messaging
