## ADR-006: Locale and Platform Signal Collection Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-006

### Context

The application needs to collect low-risk browser metadata (timezone, languages, platform, Do Not Track state) so downstream modules (Fingerprint Receipt, Zero Click OSINT, Shadow Profile) can analyze and present these signals. The `src/signals/` directory exists in the repository structure (per CLAUDE.md) but contains no modules yet. This is the first signal collector to be implemented.

### Decision

Add a single pure-function signal collector (`src/signals/locale-signals.ts`) that:

1. Defines a `LocaleSignals` interface with four typed fields
2. Exports `collectLocaleSignals()` which reads from `Intl.DateTimeFormat`, `navigator.languages`, `navigator.platform`, and `navigator.doNotTrack` with feature detection
3. Returns safe fallback strings when any API is unavailable or throws
4. Unit tests in `tests/us-006/` covering happy-path, fallback, and type-shape validation

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to an existing empty directory and one new test directory. No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. All browser APIs used are standard Web Platform APIs available in any modern browser and in Vitest's jsdom/happy-dom environment.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover the new files without modification.

**Risks identified and mitigations:**
- `navigator.platform` is deprecated but still widely supported; the fallback-to-`'unknown'` pattern handles eventual removal gracefully with zero downstream impact
- `doNotTrack` has inconsistent cross-browser implementations (`'1'`/`'0'`/`null`/`undefined`); the collector normalizes to a string without interpreting meaning, deferring semantic analysis to downstream modules
- Browser API mocking in Vitest (jsdom) may diverge from real browsers; stubbing at the global level (as planned) is the standard mitigation and sufficient for unit-level confidence
- `navigator.languages` returns a live array in some browsers; returning a copy prevents downstream mutation — low risk but good defensive practice

**Consistency with existing architecture:** Fully consistent. The plan follows CLAUDE.md conventions exactly: file lands in `src/signals/` (the designated directory), uses pure functions with feature detection, separates data collection from rendering, and tests land in `tests/us-006/`. No runtime dependencies added. The pattern established here (typed interface + pure collector + safe fallbacks) sets a clean precedent for future signal collectors.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Establishes the first signal collection module and a reusable pattern (interface + collector + fallbacks) for subsequent signal tickets
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** None significant — this is a minimal, additive change with no architectural risk
