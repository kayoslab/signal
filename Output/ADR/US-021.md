## ADR-021: Permissions API Adapter Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-021

### Context

The application needs to audit granted, denied, and prompt-state permissions so the Permission Debt module can calculate a debt score and present permission posture to the user. The `src/permissions/` directory exists in the repository structure (per CLAUDE.md) but contains only a `.gitkeep` placeholder. The `src/signals/` directory already has three collectors (US-006, US-007, US-008) that establish a clear pattern of typed interface + pure collector function + safe fallbacks + feature detection. US-021 creates the first module in `src/permissions/`, following analogous conventions.

### Decision

Add a permissions adapter (`src/permissions/permissions-adapter.ts`) that:

1. Defines a `PermissionState` union type (`'granted' | 'denied' | 'prompt' | 'unsupported'`) and a `PermissionCheckResult` interface with `name` and `state` fields
2. Implements a feature detection helper `hasPermissionsAPI()` that checks for `navigator.permissions` and `navigator.permissions.query` availability
3. Implements an internal `queryPermission(name)` async function wrapping `navigator.permissions.query()` in try-catch, returning `'unsupported'` on failure
4. Exports `checkPermissions(): Promise<PermissionCheckResult[]>` that iterates over a predefined list of standard permission names and returns results for each
5. Unit tests in `tests/us-021/` covering happy-path, fallback, mixed results, and full API absence scenarios

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to an existing empty directory (`src/permissions/`) and one new test directory (`tests/us-021/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The Permissions API is a standard Web Platform API. Test mocking uses existing Vitest capabilities (`vi.stubGlobal`) already used across US-006, US-007, and US-008 tests.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover the new files without modification.

**Risks identified and mitigations:**
- **Permission name strings vary across browsers:** Some names (e.g., `'clipboard-read'`, `'clipboard-write'`) throw `TypeError` or `DOMException` in certain browsers. The per-permission try-catch design correctly mitigates this — any throwing name returns `'unsupported'` without affecting other queries. This is the right approach.
- **Async iteration pattern:** `navigator.permissions.query()` is async; the plan iterates over a list of permission names calling an async function for each. This is straightforward and follows standard async/await patterns. Tests must properly handle promise resolution — the existing test infrastructure (Vitest with async test support) handles this natively.
- **Permissions API availability:** The API is not available in all contexts (e.g., some mobile browsers, non-secure contexts). The `hasPermissionsAPI()` guard ensures the entire module degrades to returning `'unsupported'` for all permissions when the API is absent. This matches the fallback strategy used by all signal collectors.
- **No types.ts shared file:** Unlike `src/signals/` where US-007 planned a shared `types.ts` (though it was not actually created — interfaces remain in their respective collector files), the permissions adapter defines its types inline. This is appropriate since US-021 is the only module in `src/permissions/` and there is no immediate need for a shared types file. If future permissions modules are added, types can be extracted at that point.
- **Standard permission names list maintenance:** The hardcoded list of permission names to query needs to stay aligned with browser support. This is a content concern, not an architectural one — the list is easy to update and the try-catch per-name design means adding new names carries zero risk.

**Consistency with existing architecture:** Fully consistent. The plan follows the same conventions established by the signal collectors: typed interface with union-type fallbacks, pure exported function, internal helpers, feature detection before API access, and try-catch around potentially throwing APIs. The key difference is that this module uses `async` functions (due to the Permissions API being promise-based), which is a natural extension of the pattern. Files land in `src/permissions/` per CLAUDE.md conventions. Tests land in `tests/us-021/` following the per-ticket test directory convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Establishes the first permissions module, providing the adapter layer needed by the Permission Debt dashboard module (CLAUDE.md Required Module #4)
- **Positive:** The `'unsupported'` state cleanly distinguishes between "browser says no" (`'denied'`) and "browser doesn't support this query" — important for accurate debt scoring downstream
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** Per-permission try-catch means the adapter is resilient to browser-specific quirks without needing browser detection
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** None significant — this is a minimal, additive change with no architectural risk
