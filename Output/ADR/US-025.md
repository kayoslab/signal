## ADR-025: Conservative Shadow Profile Inference Rules

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-025

### Context

The application needs bounded inference rules that derive plausible user traits from observable browser signals for the Shadow Profile module (CLAUDE.md Required Module #5). The `src/modules/shadow-profile/` directory contains `inference-schema.ts` (US-024), which exports the `InferenceStatement` interface, `createInference()` factory, `isValidInference()` guard, and the `INFERENCE_MARKER` constant. The `src/signals/snapshot.ts` (US-009) provides `SignalSnapshot` composing `locale` (`LocaleSignals`), `device` (`DeviceCapabilitySignals`), and `rendering` (`RenderingSignals`) with `MISSING` sentinels. The `src/permissions/permissions-adapter.ts` (US-021) provides `PermissionCheckResult[]` with `PermissionState` types. US-025 is the first module to produce `InferenceStatement[]` using the schema established in US-024.

### Decision

Add inference rules to the shadow profile module:

1. **Rule input type** (`src/modules/shadow-profile/inference-rules.ts`) — `RuleInput` accepting `SignalSnapshot` and optionally `PermissionCheckResult[]`, with an `applyInferenceRules(input): InferenceStatement[]` aggregator calling all individual rule functions
2. **Five individual rule functions** — each a pure function returning `InferenceStatement | null`: desktop usage pattern (hardwareConcurrency ≥ 4, no touch, wide screen), geographic region from timezone, privacy-conscious user (DNT + denied permissions), high-end hardware (threads ≥ 8, high DPR, pro GPU keywords), multilingual user (2+ languages)
3. **All rules use `createInference()` from `inference-schema.ts`** to guarantee marker and schema compliance
4. **Individual rule exports for unit testing**, aggregate `applyInferenceRules` for module consumption
5. **Barrel export update** (`src/modules/shadow-profile/index.ts`) — re-export rule functions and aggregator
6. **Tests** (`tests/us-025/inference-rules.test.ts`) — per-rule match/no-match, certainty language regex scan, evidence field validation, integration test verifying ≥ 4 rules fire simultaneously

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to the existing `src/modules/shadow-profile/` directory, modifies the barrel export (`index.ts`), and adds one new test directory (`tests/us-025/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/signals/snapshot.ts` (types and sentinels), `src/permissions/permissions-adapter.ts` (types only), and sibling `inference-schema.ts` (factory and types). No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**

- **Rule thresholds may be too narrow or broad:** The desktop rule uses `hardwareConcurrency >= 4`, `touchSupport === false`, and `screenWidth >= 1280`. These are reasonable heuristics but will miss 2-core desktops and match some tablets with keyboards. The `medium` confidence label appropriately communicates this uncertainty. Thresholds are simple integer comparisons in a pure function, trivially adjustable without architectural changes. The high-end hardware rule's GPU keyword matching (`'RTX'`, `'Pro'`, `'M1'`, `'M2'`, `'M3'`, `'Radeon Pro'`) is inherently fragile — new GPU models (M4, M5, future NVIDIA architectures) will not match. The plan assigns `low` confidence and documents the keyword list as approximate, which is the correct mitigation.

- **GPU renderer string matching uses `String.includes()` against hardcoded keywords:** This is the most fragile part of the plan. The `renderer` field from `RenderingSignals` contains driver-reported strings that vary by OS, driver version, and browser. For example, Chrome on macOS may report `'Apple M1 Pro'` while Firefox reports `'Apple GPU'`. The keyword list must be tested against real-world renderer strings. However, the `low` confidence assignment and the rule returning `null` when no keywords match means false negatives are harmless (the inference simply doesn't appear) and false positives are unlikely given the additional `hardwareConcurrency >= 8` and `devicePixelRatio >= 2` guards.

- **Privacy-conscious rule depends on optional `PermissionCheckResult[]`:** The `RuleInput` type makes permissions optional. The privacy-conscious rule must check if permissions are provided and, if not, rely solely on `doNotTrack === '1'`. If DNT alone is insufficient (the plan requires 2+ privacy signals), the rule should return `null` when permissions are absent and only DNT is active. This is correct behavior — the rule should not infer privacy consciousness from a single signal. The implementation must handle `undefined` permissions gracefully without throwing.

- **Timezone-to-region mapping is a static lookup table:** The plan maps IANA timezone prefixes (e.g., `'America/New_York'` → `'eastern North America'`) to broad geographic regions. This mapping is inherently incomplete — there are 400+ IANA timezone identifiers. The plan should map timezone prefixes (e.g., `'America/'`, `'Europe/'`, `'Asia/'`) to continental regions rather than attempting city-level mapping, and return `null` for unrecognized timezones. The `medium` confidence assignment is appropriate for this level of geographic granularity.

- **Certainty language may creep into rule statement strings:** The plan mandates a regex test (`/\b(definitely|certainly|always|never|proves|guaranteed|undoubtedly|without doubt)\b/i`) across all rule outputs. This is essential — every statement string must use hedging language ("consistent with," "suggests," "pattern is consistent with") rather than assertive language. The `createInference()` factory does not validate statement content (it only validates structure), so the test-level regex is the correct enforcement point.

- **`SignalSnapshot` field type handling:** Several snapshot fields are union types (`number | string`, `boolean | string`). The rules must perform type narrowing before numeric comparisons: `typeof snapshot.device.hardwareConcurrency === 'number'` before `>= 4`. If the value is a string sentinel (`'unknown'`/`'unavailable'`), the condition should fail and the rule should return `null`. This is standard TypeScript type narrowing, not an architectural concern, but must be implemented correctly for each rule.

- **`applyInferenceRules` must handle mixed rule results:** The aggregator collects non-null results from all rule functions. If all rules return `null` (e.g., all signals are missing sentinels), the aggregator returns an empty array. This is correct behavior — no inferences should be made when no evidence exists. Downstream consumers must handle empty arrays gracefully.

- **Barrel export expansion:** The `index.ts` barrel currently exports only from `inference-schema.ts`. Adding re-exports from `inference-rules.ts` is additive and does not affect existing imports. No existing consumer imports from the barrel (the shadow profile module has no rendering or bootstrap wiring yet — that will come in subsequent tickets).

- **No rendering or bootstrap changes:** This ticket is purely data-layer — it adds inference rule logic but does not mount anything to the DOM or modify `main.ts`. The shadow profile rendering module (a future ticket) will consume `applyInferenceRules()` to generate cards or receipt rows. This separation is correct per CLAUDE.md's mandate to separate data collection from rendering.

**Consistency with existing architecture:** Fully consistent. The module extends `src/modules/shadow-profile/` with inference logic that consumes `SignalSnapshot` (from `src/signals/`) and `PermissionCheckResult` (from `src/permissions/`) — clean cross-directory dependencies following the established layered architecture. Each rule is a small pure function per CLAUDE.md coding rules. The `createInference()` factory guarantees every output carries the `[Inference]` marker per CLAUDE.md content rules. The `RuleInput` pattern mirrors how `calculateEntropyScore()` accepts `SignalSnapshot` and `calculatePermissionDebtScore()` accepts `PermissionCheckResult[]` — typed input → typed output with no side effects. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All computation is client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The barrel export modification is additive — existing imports of `InferenceStatement`, `createInference`, `isValidInference`, `INFERENCE_MARKER`, `ConfidenceLabel`, and `EvidenceEntry` from `src/modules/shadow-profile/` continue to work unchanged.

### Consequences

- **Positive:** Delivers five bounded inference rules with observable evidence, exceeding the acceptance criteria minimum of four
- **Positive:** Every rule uses `createInference()` from the US-024 schema, guaranteeing structural compliance and `[Inference]` marker presence by construction
- **Positive:** Pure function design makes each rule independently testable with deterministic inputs/outputs
- **Positive:** Optional permissions in `RuleInput` allows rules to function with signals only, gracefully degrading when permissions data is unavailable
- **Positive:** Test-level regex scanning for certainty language provides automated enforcement of CLAUDE.md content rules
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All computation remains client-side; no trust rule violations
- **Negative:** GPU keyword matching in the high-end hardware rule is fragile and will require periodic updates as new hardware ships — mitigated by `low` confidence and the rule being non-critical
- **Negative:** Timezone-to-region mapping is inherently incomplete — mitigated by broad regional categories and `medium` confidence
- **Trade-off:** Five separate rule functions in a single file is appropriate for the current scope; if the rule set grows significantly (10+), extraction into individual files or a rules registry pattern would improve maintainability, but that refactoring is premature at this stage
