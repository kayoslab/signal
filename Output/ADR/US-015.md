## ADR-015: Privacy Posture and Uniqueness Estimate on Receipt

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-015

### Context

The application needs to display a human-readable privacy posture summary and uniqueness estimate on the fingerprint receipt, derived from the entropy score computed in US-014. The `src/scoring/entropy-score.ts` (US-014) exports `calculateEntropyScore(snapshot): EntropyResult` with a `score` field (0â€“100 integer), `maxPossible`, and `breakdown`. The `src/modules/fingerprint/fingerprint-receipt.ts` (US-013, US-017) renders a receipt with 12 signal rows plus a rerun button. The `src/ui/receipt.ts` (US-011) provides `ReceiptRow` interface (`{ label: string; value: string }`) and `createReceiptRow(label, value)`. The `src/scoring/` directory contains two scoring modules (`entropy-score.ts`, `permission-debt-score.ts`) following the pattern of typed result interfaces with pure scoring functions.

### Decision

Add a privacy posture computation module and integrate it into the fingerprint receipt:

1. **Privacy posture function** (`src/scoring/privacy-posture.ts`) â€” pure function `computePrivacyPosture(result: EntropyResult): PrivacyPostureResult` returning entropy label (`'Low' | 'Moderate' | 'High'`), uniqueness estimate text, and privacy posture summary line, using threshold constants (LOW â‰¤33, MODERATE 34â€“66, HIGH â‰Ą67)
2. **Receipt extension** (`src/modules/fingerprint/fingerprint-receipt.ts`) â€” after computing entropy, call `computePrivacyPosture()` and append three new `ReceiptRow` entries: 'Entropy Level', 'Uniqueness Estimate' (with '[Heuristic]' suffix), and 'Privacy Posture'
3. **Unit tests** (`tests/us-015/privacy-posture.test.ts`) â€” boundary conditions, label mapping, heuristic marker presence, determinism
4. **Integration tests** (`tests/us-015/fingerprint-receipt-posture.test.ts`) â€” DOM verification of new rows

### Architectural Impact Assessment

**Scope:** Very small â€” adds one source file to `src/scoring/` (third scoring module after `entropy-score.ts` and `permission-debt-score.ts`), modifies one existing file (`src/modules/fingerprint/fingerprint-receipt.ts`), and adds one new test directory (`tests/us-015/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports only the `EntropyResult` type from `src/scoring/entropy-score.ts`. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**

- **Existing US-013 test asserts exactly 12 `.receipt-row` elements:** The test at `tests/us-013/fingerprint-receipt.test.ts:91` asserts `expect(rows.length).toBe(12)`. Adding 3 new receipt rows changes the count to 15, which **will break this test**. The implementation must update the US-013 test assertion from 12 to 15. This is the highest-impact consequence of this change â€” not architecturally significant, but a concrete regression that must be addressed during implementation.

- **US-017 rerun flow must also recalculate posture:** The current `fingerprint-receipt.ts` rerun callback (lines 19â€“29) recollects the snapshot and repopulates `.receipt-rows`. The modified version must also recalculate entropy score and privacy posture on rerun, then include the three new rows in the repopulated container. If the rerun callback only regenerates the original 12 `formatSnapshotToRows` rows without posture rows, the posture data will disappear after rerun. The implementation must ensure the full row set (12 signal rows + 3 posture rows) is rebuilt on each rerun.

- **Threshold values (33/66) are arbitrary:** The plan correctly identifies this and uses named constants for easy adjustment. The thresholds map to the 0â€“100 entropy score: â‰¤33 is Low (most signals missing/blocked), 34â€“66 is Moderate, â‰Ą67 is High (most signals present and distinguishing). These are reasonable ordinal boundaries but have no empirical backing. Keeping them as exported constants enables future tuning without API changes.

- **Uniqueness estimate text (e.g., '~1 in 1,500 browsers') has no statistical backing:** The plan requires this text to always carry a '[Heuristic]' suffix, which directly satisfies the acceptance criterion "Estimate clearly labeled as heuristic" and aligns with CLAUDE.md content rules ("never claim certainty where inference is used"). The estimate text is a fixed string per band, not a calculated statistical probability â€” this is the appropriate approach given the lack of a reference dataset.

- **`computePrivacyPosture` depends on `EntropyResult` interface stability:** The `EntropyResult` interface is simple (`score`, `maxPossible`, `breakdown`) and `computePrivacyPosture` only needs the `score` field. This is a minimal, stable dependency. If `EntropyResult` is extended with optional fields in future tickets, `computePrivacyPosture` is unaffected.

- **Placement in `src/scoring/` vs `src/modules/fingerprint/`:** The plan places the posture function in `src/scoring/`, which is appropriate â€” it is a pure scoring/classification function with no DOM logic, consistent with CLAUDE.md's mandate "never mix DOM logic into scoring modules." The fingerprint receipt module orchestrates the call, keeping the separation of concerns intact.

- **Receipt row ordering:** The three new rows (Entropy Level, Uniqueness Estimate, Privacy Posture) should appear after the 12 signal rows, creating a natural flow from raw data to derived assessment. The plan's approach of appending rows to the existing array achieves this.

- **If fingerprint receipt is not yet mounted in `main.ts`:** The plan's risk section correctly notes this. The `renderFingerprintReceipt` function is exported and available but may not be wired into the app bootstrap. The posture rows will exist in the receipt element regardless of whether it is currently mounted â€” this is out of scope for US-015 and does not affect the implementation.

**Consistency with existing architecture:** Fully consistent. The privacy posture function follows the established `src/scoring/` pattern: typed result interface, pure function, threshold constants with clear semantics. The receipt extension follows the existing `fingerprint-receipt.ts` orchestration pattern: collect data â†’ compute â†’ format â†’ render. The '[Heuristic]' labeling aligns with the `INFERENCE_MARKER` pattern established in `src/modules/shadow-profile/inference-schema.ts` (US-024), though it uses inline text rather than the formal marker constant â€” appropriate since this is a receipt row value, not a structured inference statement. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All computation is client-side per trust rules.

**No breaking changes** to existing build pipeline or deployment. The US-013 row count test requires a trivial update (12 â†’ 15).

### Consequences

- **Positive:** Translates the raw entropy score (US-014) into human-readable privacy posture information, making the fingerprint receipt actionable for non-technical users
- **Positive:** Pure function in `src/scoring/` maintains clean separation between scoring logic and DOM rendering
- **Positive:** Threshold constants enable easy tuning of posture bands without code restructuring
- **Positive:** '[Heuristic]' suffix on uniqueness estimate satisfies CLAUDE.md content rules and acceptance criteria
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All computation remains client-side; no trust rule violations
- **Negative:** Breaks existing US-013 test (row count assertion 12 â†’ 15) â€” must be updated during implementation
- **Negative:** Rerun callback in `fingerprint-receipt.ts` must be extended to include posture rows, adding complexity to the rerun flow
- **Trade-off:** Fixed estimate text per band (not calculated probability) is less precise but avoids false statistical authority â€” the right trade-off given CLAUDE.md's content rules against claiming certainty
