## ADR-019: Zero Click OSINT Card Component

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-019

### Context

The application needs a reusable intelligence card component for displaying passive OSINT findings (CLAUDE.md Required Module #3). Each card must show a title, value, source, confidence level, and "why it matters" explanation. The `src/ui/` directory contains one component (`receipt.ts`, US-011) establishing the pattern: pure functions using `document.createElement`, returning `HTMLElement`, with exported TypeScript interfaces. The `src/styles/` directory has a complete design token system (`tokens.css`, US-004/US-005) including semantic color tokens (`--color-success`, `--color-warning`, `--color-danger`), a base `.card` class (`card.css`, US-005), and shared component CSS imported via `global.css`.

### Decision

Add a reusable OSINT card component with:

1. **Data interface and component** (`src/ui/osint-card.ts`) — `OsintFinding` interface with `title`, `value`, `source`, `confidence` (`'high' | 'medium' | 'low'`), and `whyItMatters` fields. Pure function `createOsintCard(finding: OsintFinding): HTMLElement` returning a semantic card element with BEM-style classes (`.osint-card`, `.osint-card-title`, `.osint-card-value`, `.osint-card-source`, `.osint-card-confidence`, `.osint-card-why`) and a `data-confidence` attribute for CSS-driven color variants.
2. **Scoped styles** (`src/styles/osint-card.css`) — extends `.card` base pattern with confidence badge color mapping via `[data-confidence]` attribute selectors using existing semantic tokens (`--color-success` for high, `--color-warning` for medium, `--color-danger` for low). Typography follows financial-report precision aesthetic using existing font tokens.
3. **Global CSS import** — `@import './osint-card.css'` added to `src/styles/global.css` after the existing `receipt.css` import.
4. **Tests** (`tests/us-019/osint-card.test.ts`) — unit tests covering all five acceptance criteria independently, confidence level attribute mapping, CSS file existence and token usage verification.

### Architectural Impact Assessment

**Scope:** Very small — adds one TypeScript file to `src/ui/` (second component after `receipt.ts`), one CSS file to `src/styles/`, modifies one line in `src/styles/global.css`, and adds one test directory (`tests/us-019/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The component uses standard DOM APIs available in Vitest's happy-dom environment (configured in US-004). No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**
- **Confidence badge color mapping must use existing semantic tokens:** Verified — `--color-success` (#22c55e), `--color-warning` (#eab308), and `--color-danger` (#ef4444) all exist in `src/styles/tokens.css`. No token creation needed. Risk eliminated.
- **CSS class naming collision with `.card` base styles:** The plan correctly uses `.osint-card` prefix throughout (BEM-style), which will not collide with the existing `.card` class in `card.css`. The component may optionally compose with `.card` (adding both classes to the root element) to inherit base card styles (background, border, radius, shadow, padding), then override/extend via `.osint-card` specificity. This is the correct CSS composition pattern, consistent with how `.receipt` relates to `.card`.
- **Reusability across multiple modules:** The `OsintFinding` interface is intentionally generic — it maps directly to the CLAUDE.md requirement for OSINT cards (finding, source, confidence, why it matters). The interface is suitable for consumption by the Zero Click OSINT module, and potentially by Shadow Profile and Threat Model modules if they need similar card layouts. The interface should remain focused on its stated contract without over-abstraction for hypothetical use cases.
- **`data-confidence` attribute approach:** Using a data attribute for CSS-driven color variants is a clean pattern that avoids conditional class generation in JavaScript. The CSS uses `[data-confidence="high"]`, `[data-confidence="medium"]`, `[data-confidence="low"]` attribute selectors. This is more maintainable than adding/removing modifier classes and is consistent with semantic HTML practices.
- **Test environment compatibility:** happy-dom (configured globally in `vitest.config.ts` per US-004) provides `document.createElement` and DOM querying. The component creates standard HTML elements with text content, classes, and data attributes — all well-supported by happy-dom. CSS file content tests use `fs.readFileSync` (same pattern as US-005 and US-011 tests). No test environment concerns.

**Consistency with existing architecture:** Fully consistent. The component follows the exact pattern established by `src/ui/receipt.ts` (US-011): exported TypeScript interface, pure function accepting typed input, returning `HTMLElement`, using `document.createElement` with class assignment. CSS follows the shared component pattern from US-005 (`card.css`, `button.css`, `receipt.css`) — placed in `src/styles/` and imported via `global.css`. This is appropriate since the OSINT card is a reusable shared component, not a module-specific style. Tests follow the `tests/us-{number}/` convention with DOM structure assertions plus CSS content verification. No runtime dependencies added. All rendering is client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Adds the second `src/ui/` component, reinforcing the established pure-function component pattern and providing the card primitive needed by the Zero Click OSINT module (CLAUDE.md Required Module #3)
- **Positive:** The `OsintFinding` interface provides a clear data contract for upstream modules that generate passive findings
- **Positive:** Confidence badge with semantic color tokens provides immediate visual risk communication aligned with CLAUDE.md's visual direction (financial-report precision, security operations center clarity)
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** `data-confidence` attribute approach is accessible to CSS and to assistive technologies via attribute selectors, supporting the project's accessibility requirements
- **Negative:** None significant — this is a minimal, additive change with no architectural risk
- **Trade-off:** Global CSS import means OSINT card styles are loaded even if the component is not used on a given view; negligible impact given the small CSS footprint and static single-page architecture
