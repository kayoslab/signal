## ADR-008: Rendering Engine Signal Collection Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-008

### Context

The application needs to collect rendering engine signals (WebGL support detection and GPU renderer string) so downstream modules (Fingerprint Receipt, Shadow Profile, Threat Model) can estimate device uniqueness and traits. The `src/signals/` directory contains `locale-signals.ts` (US-006) as the established signal collector pattern. US-008 is the third collector following the same conventions.

### Decision

Add a rendering engine signal collector (`src/signals/rendering-signals.ts`) that:

1. Defines a `RenderingSignals` interface with two fields: `webglSupported: boolean` and `renderer: string`
2. Exports a pure function `collectRenderingSignals()` that feature-detects canvas and WebGL context availability, extracts the unmasked renderer string via `WEBGL_debug_renderer_info` extension, and returns safe fallbacks when any step fails
3. Cleans up GPU resources after collection (`loseContext()` + discard canvas) to avoid memory leaks
4. Wraps all WebGL operations in try/catch to guarantee no uncaught runtime errors in restricted environments
5. Treats empty renderer strings as `'unavailable'` for downstream consistency
6. Unit tests in `tests/us-008/` covering happy-path, all fallback scenarios, and type-shape validation

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to the existing `src/signals/` directory and one new test directory (`tests/us-008/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. WebGL and Canvas are standard Web Platform APIs. No runtime or dev dependencies added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover the new files without modification.

**Risks identified and mitigations:**
- **WebGL mocking complexity in jsdom/happy-dom:** Neither test environment provides real canvas/WebGL support. The plan correctly identifies that `document.createElement` and the returned context object must be mocked manually. This is standard practice for WebGL unit testing and is low risk given the small API surface (createElement → getContext → getExtension → getParameter).
- **Empty renderer strings:** Some browsers return empty strings for the unmasked renderer even when the extension is available. The plan correctly treats empty strings as `'unavailable'`, ensuring consistent downstream behavior.
- **GPU memory leaks from context creation:** The plan includes explicit cleanup via `WEBGL_lose_context` extension and canvas disposal. This is important for repeated calls (e.g., re-run audit) and demonstrates good resource management.
- **SecurityError in restricted environments:** Private browsing modes, sandboxed iframes, and enterprise policies may cause `getContext()` to throw rather than return null. The try/catch wrapper handles this gracefully with fallback values.
- **`experimental-webgl` fallback context name:** The plan attempts `'webgl'` first, then `'experimental-webgl'` for older browser compatibility. This is a well-established pattern and adds negligible complexity.

**Consistency with existing architecture:** Fully consistent. Follows the identical pattern established by US-006 (typed interface + pure collector function + safe fallbacks + feature detection). File lands in `src/signals/` per CLAUDE.md conventions. Tests land in `tests/us-008/` following the per-ticket test directory convention. No runtime dependencies added. All data remains client-side per trust rules. Renderer collection is optional (returns fallback when unavailable), satisfying the ticket's note that "Renderer collection must be optional."

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Adds the third signal collector, further reinforcing the reusable pattern (interface + collector + fallbacks) established by US-006
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** Explicit GPU resource cleanup prevents memory leaks during repeated audits
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** None significant — this is a minimal, additive change with no architectural risk
