## ADR-020: Render Zero Click OSINT Findings

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-020

### Context

The application needs to display passive OSINT findings as intelligence cards (CLAUDE.md Required Module #3) showing what can be known about a user without any interaction. The `src/signals/snapshot.ts` (US-009) provides `collectSnapshot()` returning a frozen `SignalSnapshot` with `locale`, `device`, and `rendering` sub-objects plus `MISSING` sentinel constants. The `src/ui/osint-card.ts` (US-019) provides `OsintCardData` interface and `createOsintCardList(cards)` component for rendering cards with title, value, source, confidence, and whyItMatters fields. The `src/modules/` directory contains three established modules â€” `intro/`, `fingerprint/`, and `permission-debt/` â€” following the convention of per-module subdirectories with barrel exports, render functions, and format/inference helpers. The `src/app/main.ts` currently renders intro â†’ shell â†’ permission-debt module into `#dashboard`.

### Decision

Add a Zero Click OSINT module with:

1. **Core mapping function** (`src/modules/zero-click-osint/map-snapshot-to-findings.ts`) â€” pure function `mapSnapshotToFindings(snapshot: SignalSnapshot): OsintCardData[]` deriving 7-8 OSINT findings from raw signals: timezone, language, browser family, device class, screen profile, rendering engine, input capability, and approximate region
2. **Inference helpers** â€” three small pure functions in dedicated files: `infer-browser-family.ts` (platform + WebGL vendor/renderer â†’ browser family guess), `infer-device-class.ts` (screen dimensions + DPR + touch â†’ mobile/tablet/desktop), `infer-region.ts` (IANA timezone â†’ broad geographic region, always low confidence with '[Heuristic]' label)
3. **Module entry point** (`src/modules/zero-click-osint/zero-click-osint-module.ts`) â€” `renderZeroClickOsintModule(): HTMLElement` calling `collectSnapshot()`, passing to `mapSnapshotToFindings()`, then to `createOsintCardList()`, wrapped in a section with heading
4. **Barrel export** (`src/modules/zero-click-osint/index.ts`)
5. **Bootstrap wiring** (`src/app/main.ts`) â€” import and mount `renderZeroClickOsintModule()` into `#dashboard` alongside the existing permission-debt module
6. **Tests** (`tests/us-020/`) â€” unit tests for each inference helper and the mapping function, plus integration test for the rendered module DOM structure

### Architectural Impact Assessment

**Scope:** Small â€” adds a new directory (`src/modules/zero-click-osint/`) with six source files (mapping function, three inference helpers, module entry point, barrel export), modifies one existing file (`src/app/main.ts`), and adds one new test directory (`tests/us-020/`) with five test files. No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/signals/` and `src/ui/osint-card.ts` modules. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Bootstrap flow change:** `src/app/main.ts` currently mounts `renderPermissionDebtModule()` (async) into `#dashboard`. The plan adds `renderZeroClickOsintModule()` (synchronous â€” `collectSnapshot()` is synchronous) alongside it. Since the OSINT module is synchronous while permission-debt is async, the OSINT module can be appended immediately without awaiting. The insertion order in `main.ts` determines visual order in the dashboard. This is straightforward and additive.

**Risks identified and mitigations:**

- **Browser family inference from platform + WebGL renderer is inherently imprecise:** `navigator.platform` is deprecated and may return empty or generic strings in modern browsers. WebGL vendor/renderer strings vary by driver version and OS. The plan correctly assigns medium confidence to browser family inference and the helpers must handle `MISSING.unknown`/`MISSING.unavailable` sentinel values gracefully, returning 'Unknown' when signals are insufficient. This is an implementation concern, not an architectural risk.

- **Region inference from timezone can be misleading:** VPN users or manually-set timezones produce incorrect geographic inferences. The plan correctly mandates low confidence and a '[Heuristic]' label on all region values, satisfying the acceptance criterion. The `infer-region.ts` helper maps IANA timezone identifiers to broad regions (e.g., 'America/*' â†’ 'Eastern North America'), not specific locations, which is appropriately conservative per CLAUDE.md content rules ("never claim certainty where inference is used").

- **Device class boundaries are fuzzy:** Mobile vs tablet classification depends on arbitrary screen width thresholds. The plan uses medium confidence for device class inference, which is appropriate. Edge cases (large phones, small tablets) will produce reasonable but imperfect classifications â€” this is inherent to the heuristic and transparently labeled.

- **`OsintCardData` field mapping to acceptance criteria:** Verified â€” the `OsintCardData` interface from `src/ui/osint-card.ts` has exactly the fields required by CLAUDE.md's OSINT card specification: `title` (finding), `value` (finding value), `source` (data source), `confidence` ('low' | 'medium' | 'high'), and `whyItMatters` (why it matters). The plan's mapping function produces this exact shape. No interface changes needed.

- **Synchronous vs async module pattern:** Unlike `renderPermissionDebtModule()` which is async (due to `checkPermissions()` returning a Promise), `renderZeroClickOsintModule()` is synchronous because `collectSnapshot()` is synchronous. This means the OSINT module can be mounted without `await`. The `main.ts` modification must account for this difference â€” the OSINT module can be appended directly while permission-debt requires awaiting. Both patterns are valid and already established in the codebase.

- **Separation of concerns verification:** The plan correctly separates data derivation (`mapSnapshotToFindings`, inference helpers) from rendering (`createOsintCardList` from `src/ui/`). The module entry point orchestrates the pipeline but contains no signal collection logic or DOM construction logic â€” it delegates to the appropriate layers. This satisfies CLAUDE.md's mandate to "separate data collection from rendering" and "never mix DOM logic into scoring modules."

- **`collectSnapshot()` called in both intro and OSINT modules:** The intro module already calls `collectSnapshot()` for signal counting. The OSINT module will call it again independently. Each call creates a new frozen snapshot with a fresh `collectedAt` timestamp. This means two separate signal collection passes occur. The signal collectors are lightweight and idempotent (reading browser properties), so the performance cost is negligible. If snapshot sharing becomes desirable for consistency, a future ticket could pass the snapshot from intro to downstream modules, but this is premature optimization and not needed for US-020.

- **Test environment for integration test:** `renderZeroClickOsintModule()` calls `collectSnapshot()` which internally calls `collectRenderingSignals()` using WebGL APIs not available in happy-dom. The integration test should mock `collectSnapshot` at the module level to return a predictable snapshot, consistent with how `tests/us-023/` mocks `checkPermissions`. This is standard practice and the plan's test recommendations align with this approach.

**Consistency with existing architecture:** Fully consistent. The module follows the established subdirectory pattern (`src/modules/zero-click-osint/` with barrel export), consistent with `intro/`, `fingerprint/`, and `permission-debt/`. The render function returns `HTMLElement`, matching the pattern. The inference helpers are small pure functions per CLAUDE.md coding rules. The mapping function separates data transformation from rendering per CLAUDE.md mandate. The module composes existing upstream layers (`src/signals/` â†’ `src/modules/zero-click-osint/` â†’ `src/ui/`) in the expected dependency direction. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The `main.ts` modification is additive â€” existing intro, shell, and permission-debt rendering are preserved.

### Consequences

- **Positive:** Delivers CLAUDE.md Required Module #3, showing users what can be passively known about them through 7-8 intelligence cards with confidence labels and source attribution
- **Positive:** Follows the established module pattern (subdirectory, barrel export, render function, data helpers), reinforcing architectural consistency
- **Positive:** Composes two existing upstream modules (`signals/snapshot` + `ui/osint-card`) cleanly, validating the layered architecture design
- **Positive:** Small pure inference helpers are trivially testable and maintainable, with clear separation from DOM rendering
- **Positive:** All inference cards carry appropriate confidence labels and the region card always includes '[Heuristic]', satisfying acceptance criteria and CLAUDE.md content rules
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** Duplicate `collectSnapshot()` call (also called in intro module) â€” negligible performance impact; can be optimized via snapshot passing in a future ticket if needed
- **Negative:** Inference heuristics (browser family, device class, region) are approximations that may produce incorrect results for edge cases â€” mitigated by conservative confidence labels and explicit heuristic labeling
- **Trade-off:** Three separate inference helper files add file count but each is focused, testable, and follows CLAUDE.md's "prefer small pure functions" rule â€” the alternative of inlining all logic in the mapping function would reduce files but increase complexity per file
