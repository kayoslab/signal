## ADR-004: Responsive Application Shell Layout

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-004

### Context

The application needs a responsive layout shell (header, main content area, footer) that provides a three-column grid on desktop (≥1200px) and stacks vertically on mobile. The `src/layout/` directory exists but is empty (`.gitkeep` only). The current `src/app/main.ts` renders a placeholder string into the `#app` div. No DOM testing environment is configured — Vitest runs in the default Node environment.

### Decision

Implement a responsive shell layout with the following changes:

1. **Test environment** — install `happy-dom` (devDependency) and add `environment: 'happy-dom'` and `css: true` to `vitest.config.ts`, enabling DOM API access and CSS import resolution in all tests
2. **Layout module** (`src/layout/shell.ts`) — pure function `createShell()` returning a DocumentFragment with semantic `<header>`, `<main>`, and `<footer>` elements
3. **Scoped CSS** (`src/layout/shell.css`) — mobile-first single-column grid, with `@media (min-width: 1200px)` breakpoint switching to `grid-template-columns: 1fr 2fr 1fr` for three-column desktop layout
4. **Global overflow fix** — add `overflow-x: hidden` to the existing `body` rule in `src/styles/global.css` (single location, no duplication)
5. **Entry point integration** — replace the placeholder text in `src/app/main.ts` with shell fragment insertion
6. **Tests** (`tests/us-004/shell.test.ts`) — DOM structure assertions via happy-dom, plus static CSS content parsing to verify responsive rules and overflow placement

### Architectural Impact Assessment

**Scope:** Small-to-medium — adds one devDependency, two new source files in an existing directory, modifies three existing files (vitest.config.ts, global.css, main.ts), and adds one test directory.

**New dependencies:** `happy-dom` (devDependency only). This is a standard lightweight DOM implementation for testing. It does not affect the runtime bundle. No new runtime dependencies are introduced.

**Build pipeline change:** None. The existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new files without modification. The `build` script (with policy pre-check gate from US-003) is unaffected.

**Configuration change:** `vitest.config.ts` gains `environment: 'happy-dom'` and `css: true`. This is the highest-impact change because it affects all test files globally.

**Risks identified and mitigations:**
- **Global happy-dom environment affecting existing tests:** Existing tests (US-001, US-002, US-003) use Node.js APIs (`fs`, `path`, `child_process`, `execSync`). happy-dom adds browser globals (document, window) on top of the Node environment — it does not remove Node APIs. Tests using `node:fs` and `node:path` (architecture-policy.test.ts, build-output.test.ts) should continue to work. If any test breaks, per-file `// @vitest-environment node` overrides can restore the original environment for specific files.
- **CSS static analysis tests vs runtime rendering:** happy-dom does not support CSS Grid layout rendering or media query evaluation. The plan correctly addresses this by testing CSS rules as static text content (parsing the CSS file string) rather than attempting runtime media query evaluation. This is the right approach.
- **`overflow-x: hidden` as horizontal scroll prevention:** While effective, this approach masks rather than prevents overflow. The plan correctly uses `fr` units in the grid (not fixed widths), which inherently prevents content overflow. The `overflow-x: hidden` in global.css is a safety net, not a primary mechanism.
- **Intermediate viewport gap (768px–1200px):** The plan defines mobile (single-column) and desktop (≥1200px, three-column) but does not specify tablet behavior. This is acceptable for US-004's acceptance criteria, which only require desktop three-column and mobile vertical stacking. A future ticket could add intermediate breakpoints if needed.

**Consistency with existing architecture:** Fully consistent. Files land in `src/layout/` (the designated directory per CLAUDE.md). CSS follows the project's scoped approach with import from the TypeScript module. The shell uses vanilla TypeScript with `document.createElement` — no framework dependencies. The mobile-first responsive pattern aligns with the project's lightweight, static-hosting approach. Tests follow the `tests/us-{number}/` convention established by prior tickets.

**No breaking changes** to existing code or build pipeline. The vitest environment change is the only modification that touches existing test infrastructure, and it is additive (adds DOM globals) rather than destructive (does not remove Node APIs).

### Consequences

- **Positive:** Establishes the application's structural foundation — all future module tickets (Fingerprint Receipt, OSINT Cards, Permission Debt, etc.) will mount into the shell's `<main>` content area
- **Positive:** happy-dom environment enables DOM-level testing for all future UI tickets without per-file configuration
- **Positive:** CSS Grid with `fr` units provides flexible, responsive layout with minimal CSS
- **Positive:** Zero runtime dependencies; layout is pure vanilla TypeScript + CSS
- **Negative:** Global happy-dom environment change requires verification that all existing tests (US-001 through US-003) still pass — minor migration risk
- **Trade-off:** Static CSS text parsing tests verify rule presence but not visual correctness; Playwright E2E tests (required by CLAUDE.md before release) will provide the visual verification layer
