## ADR-013: Populate Receipt with Advanced Fingerprint Fields

**Status:** Accepted
**Date:** 2026-04-23
**Ticket:** US-013

### Context

The application needs to display additional system traits — renderer, storage capability, and screen profile — in the fingerprint receipt when available, with fallback copy for unavailable values. The `src/signals/` barrel export (US-009) provides `collectSnapshot()` returning a frozen `SignalSnapshot` with `locale`, `device`, and `rendering` sub-objects, plus `MISSING` sentinel constants. The `src/ui/receipt.ts` (US-011) provides a reusable `createReceipt(title, rows)` component accepting `ReceiptRow[]`. The `src/modules/` directory is currently empty (`.gitkeep` only). US-012 (core fingerprint receipt formatting with 6 basic fields) has an accepted ADR but has not been implemented — it planned a flat file at `src/modules/fingerprint-receipt.ts`.

### Decision

Create a fingerprint receipt module in a subdirectory (`src/modules/fingerprint/`) with:

1. **Pure formatter** (`src/modules/fingerprint/format-snapshot.ts`) — `formatSnapshotToRows(snapshot: SignalSnapshot): ReceiptRow[]` mapping 12 snapshot fields to human-readable receipt rows: Renderer, Vendor, WebGL Version, Storage (localStorage/sessionStorage/indexedDB), Screen Resolution, Device Pixel Ratio, Touch Support, CPU Threads, Timezone, Languages, Platform, Do Not Track
2. **Fallback handling** — `FALLBACK_COPY = 'Not available'` constant with `isMissing(value)` helper checking against `MISSING` sentinels, empty strings, and null
3. **Module entry point** (`src/modules/fingerprint/fingerprint-receipt.ts`) — `renderFingerprintReceipt(): HTMLElement` orchestrating `collectSnapshot()` → `formatSnapshotToRows()` → `createReceipt()`
4. **Barrel export** (`src/modules/fingerprint/index.ts`)
5. **Tests** in `tests/us-013/` — unit tests for the formatter (fully-populated, all-unavailable, mixed states) and integration test for the rendered receipt element

### Architectural Impact Assessment

**Scope:** Small — adds a new directory (`src/modules/fingerprint/`) with three source files, and one new test directory (`tests/us-013/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/signals/` and `src/ui/receipt.ts`. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new files without modification.

**Risks identified and mitigations:**

- **Scope overlap with US-012:** This is the most significant concern. US-012's accepted ADR planned a flat file at `src/modules/fingerprint-receipt.ts` with 6 basic fields (timezone, languages, platform, touch, CPU threads, DNT). US-013's plan creates a subdirectory at `src/modules/fingerprint/` and includes all 12 fields — the 6 basic fields from US-012 plus the 6 advanced fields from US-013's acceptance criteria. Since US-012 has not been implemented and US-013 does not list US-012 as a dependency, the plan effectively subsumes US-012's scope. **Mitigation:** This is acceptable if US-012 is marked as delivered by this implementation. If US-012 is implemented first (as sequencing suggests), the plan must be revised to extend rather than replace US-012's work, and the directory restructuring (flat file → subdirectory) should be acknowledged as a refactor. The implementation should verify US-012's status before proceeding.

- **Structural precedent — subdirectory vs flat file:** US-012 ADR chose a flat file because it was a single-file module. US-013's plan creates a subdirectory with 3 files (formatter, orchestrator, barrel), which is consistent with US-010's pattern (`src/modules/intro/` has count-signals.ts, intro-sequence.ts, intro-sequence.css). The subdirectory approach is the right call when the module has multiple files — but it conflicts with US-012's flat-file decision. If both tickets are implemented, this creates an inconsistency that must be resolved.

- **Storage formatting complexity:** `DeviceCapabilitySignals.storageSupport` is a nested object with three `boolean | string` fields. The plan must decide how to represent this as a single receipt row value. A comma-separated list of available stores (e.g., "localStorage, indexedDB") or "None available" is the appropriate approach. The `isMissing()` helper must handle `boolean` values (storage returns `true` when available, `'unavailable'` when not) — checking only against string sentinels would incorrectly treat `true` as missing.

- **Screen resolution formatting:** `screenWidth` and `screenHeight` are typed as `number | string` (number when available, `'unavailable'` string when not). The formatter must handle mixed states (one dimension available, other unavailable) and format as `"1920 × 1080"` when both are numbers. The `×` character should be the Unicode multiplication sign, not the letter "x", for typographic precision consistent with CLAUDE.md's visual direction.

- **`isMissing()` helper type safety:** The helper checks against `MISSING.unknown`, `MISSING.unavailable`, empty string, and null. It must also handle `undefined` for completeness, and must NOT treat `false` or `0` as missing — both are valid signal values (e.g., `touchSupport: false`, `hardwareConcurrency: 0` on some devices). The `unknown` type parameter is appropriate since snapshot fields have varied types.

- **Test environment for integration test:** `renderFingerprintReceipt()` calls `collectSnapshot()` which internally calls `collectRenderingSignals()`, which calls `document.createElement('canvas')` and `canvas.getContext('webgl')`. In Vitest's happy-dom/jsdom environment, `getContext('webgl')` returns `null`. The integration test must either mock `collectSnapshot` or accept that rendering signals will contain fallback values. Mocking the snapshot is cleaner and more deterministic — this aligns with the separation of concerns (test formatting independently from collection).

- **First module in `src/modules/` sets the pattern (if US-012 is not done first):** The directory structure, barrel export pattern, and data flow pipeline established here will be followed by US-014, US-015, and subsequent module tickets. The plan's approach (pure formatter separate from DOM orchestrator, barrel re-export) is a clean pattern consistent with CLAUDE.md's separation of concerns mandate.

- **`ReceiptRow.value` is typed as `string`:** All formatted values must be coerced to strings. Numeric values like `screenWidth` (number), `devicePixelRatio` (number), and `hardwareConcurrency` (number) need explicit `String()` or template literal conversion. Boolean values like `touchSupport` need semantic string mapping ('Supported'/'Not supported'). The formatter is the correct place for these conversions.

**Consistency with existing architecture:** Fully consistent. The module bridges `src/signals/` → `src/ui/` with a pure formatting function, following CLAUDE.md's mandate to separate data collection from rendering. The `ReceiptRow` interface from `src/ui/receipt.ts` is the contract. The formatter is a pure function with no side effects, trivially testable without DOM. The orchestrator handles DOM concerns in isolation. Files land in `src/modules/` per CLAUDE.md conventions. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Delivers advanced fingerprint fields (renderer, storage, screen profile) in the receipt, directly fulfilling US-013 acceptance criteria
- **Positive:** Pure formatter function is trivially testable without DOM dependencies, with clear separation from the rendering orchestrator
- **Positive:** Fallback copy handling (`'Not available'`) provides a consistent, user-friendly display for unavailable signals, meeting the acceptance criterion for fallback copy
- **Positive:** Subdirectory module structure with barrel export sets a clean, scalable pattern for future modules
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** Plan includes all 12 fields (6 basic from US-012 scope + 6 advanced from US-013 scope), which creates overlap with US-012 — must be coordinated to avoid redundant or conflicting implementations
- **Negative:** Subdirectory structure conflicts with US-012 ADR's flat-file decision — requires reconciliation if both tickets proceed independently
- **Trade-off:** The `isMissing()` helper introduces a centralized sentinel-checking function that could be extracted to a shared utility if other modules need it, but keeping it module-local for now avoids premature abstraction
