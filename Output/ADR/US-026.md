## ADR-026: Render Shadow Profile Module

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-026

### Context

The application needs a Shadow Profile dashboard module (CLAUDE.md Required Module #5) that displays likely assumptions third parties could make from passive browser signals. The `src/modules/shadow-profile/` directory already contains `inference-schema.ts` (US-024) with the `InferenceStatement` interface, `createInference()` factory, `isValidInference()` guard, and `INFERENCE_MARKER` constant, plus `inference-rules.ts` (US-025) with five individual rule functions and `applyInferenceRules()` aggregator consuming `SignalSnapshot` and optional `PermissionCheckResult[]`. The `src/ui/` directory contains three components (`receipt.ts`, `rerun-button.ts`, `osint-card.ts`) establishing the pattern: pure functions using `document.createElement`, returning `HTMLElement`, with exported TypeScript interfaces. The `src/app/main.ts` currently mounts two modules into `#dashboard`: `renderZeroClickOsintModule()` (synchronous) and `renderPermissionDebtModule()` (async).

### Decision

Add a Shadow Profile module with:

1. **Inference card UI component** (`src/ui/inference-card.ts`) — `createInferenceCard(data: InferenceStatement): HTMLElement` and `createInferenceSection(heading: string, cards: InferenceStatement[]): HTMLElement`, following the `createOsintCard`/`createOsintSection` pattern from `src/ui/osint-card.ts`. Each card displays: prominent `[Inference]` marker, statement text, evidence list (signal, value, source per entry), and confidence badge (low/medium/high).
2. **Inference card CSS** (`src/styles/inference-card.css`) — scoped styles for the `[Inference]` label, evidence list, confidence badges (reusing `--color-success`/`--color-warning`/`--color-danger` tokens), and responsive card grid. Imported via `global.css`.
3. **Module render function** (`src/modules/shadow-profile/shadow-profile-module.ts`) — `renderShadowProfileModule(): HTMLElement` calling `collectSnapshot()`, `applyInferenceRules()`, and `createInferenceSection()`. Handles empty state gracefully.
4. **Barrel export update** (`src/modules/shadow-profile/index.ts`) — re-export `renderShadowProfileModule`.
5. **Bootstrap wiring** (`src/app/main.ts`) — mount `renderShadowProfileModule()` into `#dashboard` after the permission-debt module.
6. **Tests** (`tests/us-026/shadow-profile-module.test.ts`) — DOM structure, `[Inference]` marker visibility, evidence display, confidence badges, empty state, certainty language absence.

### Architectural Impact Assessment

**Scope:** Small — adds two new source files (`src/ui/inference-card.ts`, `src/modules/shadow-profile/shadow-profile-module.ts`), one new CSS file (`src/styles/inference-card.css`), modifies two existing files (`src/modules/shadow-profile/index.ts`, `src/app/main.ts`), modifies one line in `src/styles/global.css`, and adds one new test directory (`tests/us-026/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/signals/`, `src/modules/shadow-profile/inference-rules.ts`, and the new `src/ui/inference-card.ts` component. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Bootstrap flow change:** `src/app/main.ts` currently mounts `renderZeroClickOsintModule()` (synchronous) and `renderPermissionDebtModule()` (async) into `#dashboard`. The plan adds `renderShadowProfileModule()` after permission-debt. Since `collectSnapshot()` is synchronous and `applyInferenceRules()` is synchronous, the shadow profile module is synchronous — it can be appended directly without `await`, consistent with the zero-click-osint module pattern. The insertion order (zero-click → permission-debt → shadow-profile) determines visual order in the dashboard, which aligns with CLAUDE.md's Required Modules order (Module #3 → #4 → #5).

**Risks identified and mitigations:**

- **Inference card component vs OSINT card reuse:** The plan creates a new `createInferenceCard` component rather than reusing `createOsintCard`. This is correct — `InferenceStatement` has a different shape than `OsintCardData` (it has `evidence: EvidenceEntry[]` with signal/value/source entries, a `marker` field, and `statement` instead of `title`/`value`/`whyItMatters`). The data contracts are semantically different: OSINT cards show observed facts, inference cards show derived assumptions. A shared component would require awkward field mapping and lose the explicit `[Inference]` marker enforcement. Following the `osint-card.ts` pattern (not reusing the component itself) is the right approach.

- **CSS class naming collision with osint-card styles:** The plan uses `.inference-card` prefix (BEM-style), which will not collide with `.osint-card` classes. Both components compose with the base `.card` class from `card.css`. The confidence badge styles in `inference-card.css` should use `.inference-card-confidence--{level}` rather than reusing `.osint-card-confidence--{level}` to maintain proper scoping. Verified: existing semantic color tokens (`--color-success`, `--color-warning`, `--color-danger`) are available in `tokens.css`.

- **`collectSnapshot()` called redundantly:** The shadow profile module will call `collectSnapshot()` independently, adding to calls already made by the intro module and zero-click-osint module. Each call creates a new frozen snapshot with a fresh `collectedAt` timestamp. The signal collectors are lightweight and idempotent (reading browser properties), so the performance cost is negligible. If snapshot sharing becomes desirable, a future ticket could pass the snapshot from upstream, but this is premature optimization.

- **`applyInferenceRules` called without permissions:** The plan calls `applyInferenceRules({ snapshot })` without passing `PermissionCheckResult[]`. This means the `inferPrivacyConscious` rule can only check `doNotTrack` and not denied permissions. Since the rule requires 2+ privacy signals, it will only fire when DNT is enabled AND other non-permission-based signals match. The plan's risk section correctly identifies this — the module produces fewer but more conservative inferences, which aligns with CLAUDE.md's content rule of not exaggerating risk. Passing permissions would require making the module async (to call `checkPermissions()`), which is an unnecessary complexity increase for marginal benefit. The permission-debt module already handles permission display.

- **Empty state handling:** If no inference rules fire (e.g., all signals are missing sentinels), `applyInferenceRules()` returns an empty array. The module must render a graceful empty-state message rather than an empty container. This is an implementation concern, not an architectural risk — the plan explicitly identifies it.

- **Module registration order in main.ts:** The plan inserts shadow profile after permission-debt, which follows CLAUDE.md's Required Modules ordering (Module #5 after #4). The current `main.ts` structure is: intro → shell → zero-click-osint (sync) → permission-debt (async). Adding shadow profile after the `await renderPermissionDebtModule()` call ensures it appears below permission debt in the dashboard. Since shadow profile is synchronous, it can be a simple `dashboard.appendChild(renderShadowProfileModule())` matching the zero-click-osint pattern.

- **No certainty language in rendered output:** All inference statements are generated by `createInference()` which enforces the `[Inference]` marker, but does not validate statement text content. US-025 tests include a certainty language regex scan, but rendered output should also be scanned. The plan's test recommendations correctly include a regex check for certainty language in rendered text, providing defense-in-depth at the UI layer.

- **Test environment for integration test:** `renderShadowProfileModule()` calls `collectSnapshot()` which internally calls `collectRenderingSignals()` using WebGL APIs not available in happy-dom. The integration test should mock `collectSnapshot` at the module level to return a predictable snapshot, consistent with how existing module tests handle this (e.g., `tests/us-020/` and `tests/us-023/`).

- **Inference card CSS placement in `src/styles/` vs co-located:** The plan places `inference-card.css` in `src/styles/` and imports it via `global.css`, following the pattern of `osint-card.css`, `card.css`, `button.css`, and `receipt.css` for reusable shared components. This is correct — the inference card is a reusable UI primitive in `src/ui/`, not a module-specific style.

**Consistency with existing architecture:** Fully consistent. The module follows the established subdirectory pattern (`src/modules/shadow-profile/` with barrel export), extending the existing directory that already contains `inference-schema.ts` and `inference-rules.ts`. The render function returns `HTMLElement` synchronously, matching the `renderZeroClickOsintModule()` pattern. The inference card component follows the exact `src/ui/` component pattern (pure function accepting typed input, returning `HTMLElement`, using `document.createElement` with class assignment). CSS follows the shared component pattern from `src/styles/`. The module composes existing upstream layers (`src/signals/` → `src/modules/shadow-profile/inference-rules` → `src/ui/inference-card`) in the expected dependency direction. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The `main.ts` modification is additive — existing intro, shell, zero-click-osint, and permission-debt rendering are preserved. The `index.ts` barrel export modification is additive — existing imports continue to work unchanged.

### Consequences

- **Positive:** Delivers CLAUDE.md Required Module #5, showing users likely third-party assumptions with explicit inference labeling, directly fulfilling all four acceptance criteria
- **Positive:** Follows the established module and UI component patterns, reinforcing architectural consistency across `src/modules/`, `src/ui/`, and `src/styles/`
- **Positive:** Composes two existing upstream layers (inference rules from US-024/US-025 + signal snapshot from US-009) cleanly, validating the layered architecture design
- **Positive:** The `createInferenceCard` component is purpose-built for the `InferenceStatement` interface, ensuring the `[Inference]` marker is always prominently displayed by construction
- **Positive:** Synchronous module pattern avoids unnecessary async complexity, consistent with zero-click-osint
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** `collectSnapshot()` is called redundantly (also called by intro and zero-click-osint modules) — negligible performance impact; can be optimized via snapshot passing in a future ticket if needed
- **Negative:** `inferPrivacyConscious` rule produces fewer results without permissions data — acceptable trade-off to keep the module synchronous and avoid duplicating permission-debt concerns
- **Trade-off:** Creating a new `createInferenceCard` component rather than reusing `createOsintCard` adds a new UI component, but the semantic difference between observed findings and derived inferences justifies the separation — inference cards must prominently display the `[Inference]` marker and evidence chain, which the OSINT card layout does not accommodate
