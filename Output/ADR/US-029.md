## ADR-029: Hardening Recommendation Rules Engine

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-029

### Context

The application needs actionable hardening recommendations generated from browser signal findings for the Hardening Actions module (CLAUDE.md Required Module #7). The `src/modules/hardening/` directory does not yet exist. The plan depends on six upstream modules: `src/signals/snapshot.ts` (US-009) providing `SignalSnapshot`, `src/permissions/permissions-adapter.ts` (US-021) providing `PermissionCheckResult[]`, `src/scoring/entropy-score.ts` (US-014) providing `EntropyResult`, `src/scoring/permission-debt-score.ts` (US-022) providing `PermissionDebtResult`, `src/modules/shadow-profile/inference-schema.ts` (US-024) providing `InferenceStatement[]`, and `src/modules/threat-model/threat-schema.ts` (US-027) providing `ThreatFinding[]` and `ThreatInput`. All upstream interfaces are implemented and stable.

### Decision

Add a hardening recommendation rules engine with:

1. **Schema definitions** (`src/modules/hardening/hardening-schema.ts`) — `HardeningRecommendation` interface with `id`, `title`, `description`, `actionSteps: string[]`, `expectedOutcome`, `relatedFindings: string[]`, `source`, `difficulty: 'Easy' | 'Medium' | 'Hard'`. `HardeningInput` interface extending `ThreatInput` shape with an additional `threats: ThreatFinding[]` field. Factory function `createRecommendation()` for consistent construction.
2. **Rules engine** (`src/modules/hardening/hardening-rules.ts`) — pure function `evaluateHardeningRules(input: HardeningInput): HardeningRecommendation[]` executing seven individual rule functions, each returning `HardeningRecommendation | null`, following the `evaluateThreatRules` pattern from US-027.
3. **Seven hardening rules** — revoke unused high-risk permissions, enable Do Not Track, reduce fingerprint surface, review notification permissions, isolate browsing profiles, clear persistent storage, audit permission debt.
4. **Barrel export** (`src/modules/hardening/index.ts`) — re-export schema types and `evaluateHardeningRules`.
5. **Unit tests** (`tests/us-029/hardening-rules.test.ts`) — per-rule tests with factory helpers reused from US-027 test patterns.

### Architectural Impact Assessment

**Scope:** Small — adds a new directory (`src/modules/hardening/`) with three source files (schema, rules engine, barrel export) and one new test directory (`tests/us-029/`). No new dependencies. No build pipeline changes. No configuration changes. No modifications to existing files.

**New dependencies:** None. The module imports types from existing `src/signals/snapshot.ts`, `src/permissions/permissions-adapter.ts`, `src/scoring/entropy-score.ts`, `src/scoring/permission-debt-score.ts`, `src/modules/shadow-profile/inference-schema.ts`, and `src/modules/threat-model/threat-schema.ts`. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new files without modification.

**No bootstrap/main.ts changes:** This ticket is purely data-layer — it adds rule evaluation logic but does not mount anything to the DOM or modify `src/app/main.ts`. A future rendering ticket will consume `evaluateHardeningRules()` to build the Hardening Actions dashboard module. This separation is correct per CLAUDE.md's mandate to separate data collection from rendering and to "never mix DOM logic into scoring modules."

**Risks identified and mitigations:**

- **`HardeningInput` is the widest input contract in the codebase:** It composes `ThreatInput` (itself composing five upstream interfaces: `SignalSnapshot`, `PermissionCheckResult[]`, `EntropyResult`, `PermissionDebtResult`, `InferenceStatement[]`) plus `ThreatFinding[]`. This creates a six-interface composite. All upstream interfaces are implemented, tested, and consumed by at least one other module. The risk of interface breakage is low — TypeScript structural typing allows additive changes. The wider concern is that the rules engine caller (future rendering module) must orchestrate all data sources, but the existing `renderThreatModelModule()` already demonstrates this five-source orchestration pattern, and adding `evaluateThreatRules()` output as the sixth source is a natural extension. Architecturally acceptable.

- **Dependency direction creates coupling from hardening to threat-model:** `HardeningInput` includes `threats: ThreatFinding[]`, which imports from `src/modules/threat-model/threat-schema.ts`. This means `src/modules/hardening/` depends on `src/modules/threat-model/` — a cross-module dependency within `src/modules/`. This is architecturally acceptable because: (1) the dependency is on a type definition file (schema), not on evaluation logic, (2) the dependency direction is acyclic (hardening → threat-model, never the reverse), and (3) the pattern is consistent with the existing `threat-model → shadow-profile` dependency where `ThreatInput` imports `InferenceStatement` from `src/modules/shadow-profile/inference-schema.ts`. The dependency is read-only type consumption, not behavioral coupling.

- **Rule thresholds must align with threat-model thresholds to avoid contradictions:** The plan uses entropy ≥ 70 for the "reduce fingerprint surface" rule, which matches the identity-exposure threat rule threshold (also ≥ 70). Permission debt ≥ 40 for the "audit permission debt" rule matches the permission-abuse threat rule threshold (also ≥ 40). The inference count ≥ 2 for "isolate browsing profiles" matches the social-engineering threat rule (also ≥ 2 medium/high inferences). This alignment is correct — hardening recommendations should trigger at the same thresholds as the threats they address. Any future threshold changes in threat-rules.ts should be mirrored in hardening-rules.ts. This coupling is acceptable because the two modules serve complementary purposes (diagnosis vs. remedy) and their thresholds should be logically consistent.

- **Hardcoded browser-specific instructions in actionSteps:** The plan correctly identifies that steps like "Go to chrome://settings" would not apply to all browsers. Using generic wording ("Open your browser's site settings" rather than browser-specific paths) is the correct approach. This is a content concern validated at the test level, not an architectural risk.

- **Schema design may need revision for the rendering ticket:** The `HardeningRecommendation` interface is intentionally focused on data — no rendering hints, no DOM references. The `difficulty` field (`'Easy' | 'Medium' | 'Hard'`) provides a classification that the rendering layer can use for visual treatment (badges, sorting, grouping) without the schema dictating how. The `actionSteps: string[]` array provides ordered steps without prescribing their visual presentation. This is the right level of abstraction for a data-layer ticket. If the rendering ticket needs additional fields (e.g., icon hints, priority ordering), they can be added as optional fields to the interface without breaking existing consumers.

- **Seven rules exceed the acceptance criteria minimum of five:** The plan provides seven rules, exceeding the "at least five possible actions" acceptance criterion. This provides coverage across all three required response categories: permissions findings (rules 1, 4, 7), uniqueness/entropy findings (rules 2, 3), and profile separation opportunities via inferences (rule 5). Rule 6 (clear persistent storage) bridges permissions and device capabilities. This is appropriate scope — each rule is a small pure function and the marginal complexity of seven vs five rules is negligible.

- **Factory helpers can be reused from US-027 tests:** The `tests/us-027/threat-rules.test.ts` file contains factory helpers (`makeSnapshot`, `makeEntropy`, `makePermissions`, `makeDebt`, `makeInference`, `makeInput`) that construct all components of `ThreatInput`. The US-029 tests need these same helpers plus a `makeThreatFinding()` helper for the additional `threats` field. Rather than importing from `tests/us-027/` (which would create a test-to-test dependency), the plan should duplicate or extract the helpers. Duplicating is acceptable given the helpers are small (< 100 lines total) and the test-to-test coupling would be more brittle than the duplication. Alternatively, shared test utilities could be extracted to a `tests/helpers/` directory, but this refactoring is premature for the current scope.

- **Content language in recommendation descriptions and actionSteps:** Per CLAUDE.md content rules ("never exaggerate risk," "avoid fear language," "use concise executive-grade wording"), all recommendation text must use measured, actionable language. For example: "Review and revoke camera and microphone permissions for sites that no longer need them" rather than "Your camera is dangerously exposed to hackers." Test-level regex scanning for certainty/fear language (consistent with US-025 and US-027 test patterns) provides automated enforcement.

- **Rule evaluation follows the established pattern exactly:** The plan uses the same structure as `evaluateThreatRules`: individual rule functions returning `T | null`, aggregator collecting non-null results. However, the plan describes each rule as "a pure function" rather than the `ThreatRule` object pattern (with `id`, `title`, `category`, `explanation`, `evaluate` method) used in `threat-rules.ts`. Either approach works — the plan's pure function approach is simpler (no rule object boilerplate) while the threat-rules object pattern is more self-documenting (explanation field for explainability). Given that CLAUDE.md requires "all heuristics must be explainable," the implementation should include explanation metadata for each rule, whether via the object pattern or via JSDoc/exported documentation. This is an implementation detail, not an architectural concern.

**Consistency with existing architecture:** Fully consistent. The module follows established conventions: per-module subdirectory in `src/modules/`, typed interfaces for data contracts, pure functions with no side effects, separation of data/scoring logic from DOM rendering. The rule-evaluation pattern mirrors `evaluateThreatRules()` from `src/modules/threat-model/threat-rules.ts` (US-027): input → array of recommendations, each rule returning `T | null`, aggregator filtering nulls. The `HardeningRecommendation` result interface with `difficulty` and `actionSteps` fields is structurally analogous to `ThreatFinding` with `severity` and `evidence`. The `HardeningInput` extending `ThreatInput` with `threats` follows the natural data pipeline: signals → scoring → inference → threats → hardening. Files land in `src/modules/hardening/` per CLAUDE.md conventions (hardening actions is Required Module #7). Tests land in `tests/us-029/` following the per-ticket convention. No runtime dependencies added. All computation is client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. This ticket is purely additive — no existing files are modified.

### Consequences

- **Positive:** Delivers seven actionable hardening rules responding to permissions findings (rules 1, 4, 7), uniqueness/entropy findings (rules 2, 3), and profile separation opportunities (rule 5), exceeding the minimum of five and directly fulfilling all four US-029 acceptance criteria
- **Positive:** Pure-function rule design makes each rule independently testable with deterministic inputs/outputs, consistent with the threat-rules pattern from US-027
- **Positive:** `createRecommendation()` factory ensures consistent `HardeningRecommendation` construction across all rules, preventing missing fields
- **Positive:** Threshold alignment with threat-model rules (entropy ≥ 70, debt ≥ 40, inferences ≥ 2) ensures hardening recommendations are logically consistent with identified threats
- **Positive:** Purely additive — zero risk to existing modules, tests, or build pipeline
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All computation remains client-side; no trust rule violations
- **Negative:** `HardeningInput` composite type creates the widest input contract in the codebase (six upstream interfaces via ThreatInput + ThreatFinding[]) — mitigated by all interfaces being stable and the consumption being read-only
- **Negative:** Cross-module dependency from hardening to threat-model creates coupling — mitigated by the dependency being on type definitions only (acyclic, read-only) and consistent with the existing threat-model → shadow-profile pattern
- **Negative:** Threshold values duplicated between hardening-rules and threat-rules could drift — mitigated by named constants and alignment being verified in tests; could be extracted to shared constants in a future refactor if the rule count grows
- **Trade-off:** Seven rule functions in a single file is appropriate for current scope; if rules grow beyond ~10, extraction into individual files or a registry pattern would improve maintainability, but premature at this stage
- **Trade-off:** Duplicating test factory helpers from US-027 rather than extracting to shared test utilities avoids test-to-test coupling at the cost of ~100 lines of duplication — acceptable given the helpers are simple and the shared extraction would be premature
