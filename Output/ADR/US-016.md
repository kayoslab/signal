## ADR-016: Add Receipt Copy Action

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-016

### Context

The application needs a copy button on the Fingerprint Receipt so users can copy receipt text to the clipboard for sharing or documentation. The `src/modules/fingerprint/fingerprint-receipt.ts` (US-012, US-013, US-015, US-017) renders a receipt with 15 signal+posture rows, a rerun button in a `.receipt-actions` container, and `aria-live="polite"` on the rows container. The `src/ui/rerun-button.ts` (US-017) establishes the button factory pattern: pure function accepting a callback, returning `HTMLButtonElement`, with double-click guard, disabled/text/class state management, and try/catch/finally for resilience. The `src/styles/receipt.css` already defines `.receipt-actions` with flexbox layout and gap spacing. The `src/ui/receipt.ts` (US-011) exports the `ReceiptRow` interface (`{ label: string; value: string }`).

### Decision

Add a receipt copy action with:

1. **Copy button factory** (`src/ui/copy-button.ts`) ã `createCopyButton(getReceiptText: () => string): HTMLButtonElement` following the `createRerunButton` pattern: `btn btn-secondary receipt-copy-btn` classes, `running` flag double-click guard, async handler with disabled/text/class toggles, try/catch/finally for resilience
2. **Clipboard interaction** ã check `navigator.clipboard?.writeText` availability; if available, call `writeText(text)` and show success feedback ('Copied!' + `receipt-copy-btn--success` class); if unavailable or rejected, show failure feedback ('Copy unavailable' + `receipt-copy-btn--fail` class); revert after 2 seconds via `setTimeout`
3. **Accessibility** ã `aria-live="assertive"` on a visually-hidden status span adjacent to the button for screen reader copy announcements
4. **CSS states** (`src/styles/receipt.css`) ã `.receipt-copy-btn--success` and `.receipt-copy-btn--fail` with subtle color shifts using existing semantic design tokens (`--color-success` for success, `--color-text-muted` for fail), with `prefers-reduced-motion` support
5. **Text formatter** (`src/modules/fingerprint/format-receipt-text.ts`) ã `formatReceiptText(rows: ReceiptRow[]): string` joining rows into a plain-text receipt block with header line ('ã Fingerprint Receipt ã') and formatted label-value pairs
6. **Receipt wiring** (`src/modules/fingerprint/fingerprint-receipt.ts`) ã import `createCopyButton`, call with a closure invoking `formatReceiptText` on current rows, prepend to `.receipt-actions` container before the rerun button
7. **Unit tests** (`tests/us-016/copy-button.test.ts`, `tests/us-016/format-receipt-text.test.ts`) ã button rendering, clipboard success/failure/unavailable paths, double-click guard, feedback revert, text formatting with sample rows, empty rows, special characters

### Architectural Impact Assessment

**Scope:** Very small ã adds two new source files (`src/ui/copy-button.ts`, `src/modules/fingerprint/format-receipt-text.ts`), modifies two existing files (`src/modules/fingerprint/fingerprint-receipt.ts`, `src/styles/receipt.css`), and adds one new test directory (`tests/us-016/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The Clipboard API is a standard Web Platform API. The button factory uses standard DOM APIs available in Vitest's happy-dom environment. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**

- **Clipboard API requires secure context (HTTPS or localhost):** The plan correctly identifies this as a risk. `navigator.clipboard` is undefined in non-secure contexts. The availability check (`navigator.clipboard?.writeText`) handles this gracefully ã the button shows 'Copy unavailable' feedback rather than throwing. This is the correct defensive approach and directly satisfies the "failure feedback shown when unavailable" acceptance criterion.

- **Clipboard permissions may be denied by browser policy:** Some browsers require user gesture context for `clipboard.writeText()`, and enterprise policies may block clipboard access entirely. The plan wraps `writeText` in try/catch, catching `DOMException` (permission denied) and showing failure feedback. This is the standard mitigation pattern, consistent with the error handling in `createRerunButton`.

- **Timer-based feedback revert (setTimeout 2s) and component lifecycle:** If the receipt is removed from the DOM before the timeout fires, the `setTimeout` callback will execute on a detached element. In a static SPA where the receipt persists for the page lifetime, this is a non-issue. The plan correctly identifies this as low risk. No cleanup mechanism is needed for the current architecture.

- **Receipt rows must stay synchronized between rendering and text formatting:** The `formatReceiptText` function receives `ReceiptRow[]` ã the same array used to build the DOM. The plan's wiring in `fingerprint-receipt.ts` passes the current `rows` array (which includes both signal rows from `formatSnapshotToRows` and posture rows from `buildPostureRows`) to `formatReceiptText` via a closure. This ensures the copied text matches the displayed content. The closure captures the rows variable, so after a rerun, the closure must reference the freshly computed rows. **Implementation note:** The closure must capture the rows dynamically (e.g., by reading from the DOM or by maintaining a mutable reference) rather than closing over the initial `rows` const, otherwise the copy button will always copy the initial data even after rerun. The plan addresses this by having `getReceiptText` be a closure that invokes `formatReceiptText` at call time ã the implementation must ensure the rows reference is updated on rerun.

- **Rerun callback must update rows reference for copy button:** The current `fingerprint-receipt.ts` rerun callback (lines 32-42) recollects the snapshot and repopulates `.receipt-rows`. After rerun, the copy button's closure must have access to the fresh rows. The simplest approach is to maintain a `let currentRows` variable in the `renderFingerprintReceipt` scope that is reassigned on each rerun, with the copy button's closure reading `currentRows` at invocation time. This is a minor implementation concern, not an architectural risk.

- **Existing US-013 and US-015 test compatibility:** US-013 tests assert receipt row counts (updated to 15 in US-015). The copy button is added to `.receipt-actions` (a sibling of `.receipt-rows`), not inside `.receipt-rows`. The row count assertion queries `.receipt-row` elements inside `.receipt-rows`, so the copy button does not affect the count. **No test regression.** US-017 tests assert the rerun button exists in `.receipt-actions` ã adding a sibling copy button does not affect these assertions. **No test regression.**

- **Button placement ã prepend before rerun:** The plan prepends the copy button before the rerun button in `.receipt-actions`. This places "Copy" on the left and "Re-run Audit" on the right in the flexbox layout (since `.receipt-actions` uses `justify-content: flex-end`). This ordering is appropriate ã copy is a more common action than rerun and should be more accessible. With the existing `gap: var(--spacing-sm)` in `.receipt-actions`, two buttons will be properly spaced.

- **`aria-live="assertive"` for copy status vs `aria-live="polite"` on receipt rows:** The receipt rows container uses `polite` (from US-017) for content updates after rerun. The copy status span uses `assertive` because clipboard feedback is a direct response to user action and should interrupt. Using `assertive` on a visually-hidden span rather than on the button itself avoids disrupting the button's accessible name. This is the correct pattern.

- **CSS token usage for feedback states:** The plan uses `--color-success` for the success state and `--color-text-muted` for the fail state. Verified ã both tokens exist in `src/styles/tokens.css`. `--color-success` (#22c55e) provides clear positive feedback; `--color-text-muted` for failure is appropriately subdued (not alarming). This aligns with CLAUDE.md's visual direction of muted, professional aesthetics. The `prefers-reduced-motion` override ensures no animated color transitions for users who prefer reduced motion.

- **`formatReceiptText` is a pure function with no DOM dependency:** The formatter takes `ReceiptRow[]` and returns a string. This separation means it can be unit tested without happy-dom, and it can be reused if other modules need plain-text receipt output (e.g., for a future share-via-email feature). The placement in `src/modules/fingerprint/` is correct ã it is fingerprint-receipt-specific formatting, not a general utility.

- **Double-click guard pattern:** The plan replicates the `running` flag pattern from `createRerunButton`. During the ~2s feedback display period, the button shows feedback text and the `running` flag prevents re-invocation. After the timeout reverts the button, it becomes clickable again. This prevents rapid clipboard writes and avoids confusing feedback state transitions. Consistent with the established pattern.

**Consistency with existing architecture:** Fully consistent. The copy button factory follows the exact pattern established by `createRerunButton`: pure function accepting typed input, returning `HTMLButtonElement`, with state management via class toggles and disabled property. CSS additions to `src/styles/receipt.css` follow the existing pattern of receipt-specific styles with BEM modifier classes. The text formatter follows CLAUDE.md's mandate to separate data logic from rendering ã it produces a string from data, not DOM. The wiring in `fingerprint-receipt.ts` follows the existing pattern of creating buttons via factory functions and appending to `.receipt-actions`. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Delivers clipboard copy functionality for the fingerprint receipt, directly fulfilling all four US-016 acceptance criteria (button visible, text copied, success feedback, failure feedback)
- **Positive:** Follows the established button factory pattern from `createRerunButton`, maintaining architectural consistency in `src/ui/`
- **Positive:** Pure `formatReceiptText` function is independently testable and reusable for future text-based sharing features
- **Positive:** `aria-live="assertive"` on status span provides immediate screen reader feedback for clipboard operations, meeting CLAUDE.md accessibility requirements
- **Positive:** Graceful degradation when Clipboard API is unavailable (non-secure context, browser policy) ensures the feature never throws unhandled errors
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations ã clipboard write is a local browser operation
- **Negative:** Timer-based feedback revert (2s setTimeout) has no cleanup on component removal ã acceptable for static SPA architecture where the receipt persists for page lifetime
- **Negative:** Copy button closure must reference fresh rows after rerun ã requires mutable `currentRows` variable in `renderFingerprintReceipt` scope, adding minor state management complexity
- **Trade-off:** `aria-live="assertive"` may interrupt screen reader users mid-task if they trigger copy accidentally ã mitigated by the double-click guard and the feedback being a brief, informative message ('Copied!' or 'Copy unavailable') rather than verbose content
