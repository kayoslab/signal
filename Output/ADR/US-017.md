## ADR-017: Receipt Rerun Action

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-017

### Context

The application needs a "Re-run Audit" button on the Fingerprint Receipt so users can refresh signal data without reloading the page. The `src/modules/fingerprint/fingerprint-receipt.ts` (US-013) currently exports `renderFingerprintReceipt(): HTMLElement` which calls `collectSnapshot()`, formats rows via `formatSnapshotToRows()`, and passes them to `createReceipt()`. The receipt component (`src/ui/receipt.ts`, US-011) builds a `.receipt` container with `.receipt-rows` children. Shared button styles (`.btn`, `.btn-secondary`) exist in `src/styles/button.css` (US-005). No button component exists in `src/ui/` — only `receipt.ts`. The `src/signals/snapshot.ts` (US-009) returns a new frozen `SignalSnapshot` on each `collectSnapshot()` call.

### Decision

Add a receipt rerun action with:

1. **Rerun button factory** (`src/ui/rerun-button.ts`) — `createRerunButton(onRerun: () => Promise<void>): HTMLElement` creating a `<button class="btn btn-secondary receipt-rerun-btn">Re-run Audit</button>` with click handler that disables the button, swaps text to a loading label, calls the async callback, then restores the button.
2. **Loading state CSS** (`src/styles/receipt.css`) — `.receipt-actions` container (flex, right-aligned), `.receipt-rerun-btn--loading` with subtle pulse animation, and `@media (prefers-reduced-motion: reduce)` override.
3. **Receipt refactor** (`src/modules/fingerprint/fingerprint-receipt.ts`) — `renderFingerprintReceipt()` enhanced to append a `.receipt-actions` div with the rerun button. The rerun callback calls `collectSnapshot()`, `formatSnapshotToRows()`, clears and repopulates `.receipt-rows`. A minimum ~300ms delay ensures the loading state is perceptible.
4. **Accessibility** — `aria-live="polite"` on `.receipt-rows` container for screen reader announcement of updated content.
5. **Unit tests** (`tests/us-017/rerun-button.test.ts`) — button rendering, callback invocation, disabled state during async execution, text change during loading, re-enable after resolve.
6. **Integration tests** (`tests/us-017/fingerprint-rerun.test.ts`) — rerun button presence, receipt row update, loading state visibility, aria-live attribute.

### Architectural Impact Assessment

**Scope:** Small — adds one new file to `src/ui/` (second component after `receipt.ts`), modifies two existing files (`src/modules/fingerprint/fingerprint-receipt.ts` and `src/styles/receipt.css`), and adds one new test directory (`tests/us-017/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The button factory uses standard DOM APIs available in Vitest's happy-dom environment (configured in US-004). Existing `.btn` and `.btn-secondary` styles from `src/styles/button.css` (US-005) are reused. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**

- **Signal collection is near-instant, loading state may flash:** `collectSnapshot()` is synchronous and fast. The plan correctly mitigates this with a minimum ~300ms artificial delay so the loading state is perceptible to users. This is a UX decision, not an architectural concern.

- **Frozen `Object.freeze()` snapshot immutability:** Each `collectSnapshot()` call returns a new frozen object, so the rerun flow creates a fresh snapshot rather than mutating an existing one. No mutation risk.

- **DOM focus loss during rerun:** If the rerun button were inside `.receipt-rows`, clearing and repopulating would destroy the focused button. The plan correctly places the button in a separate `.receipt-actions` container outside `.receipt-rows`, preserving focus during rerun. This is the right design.

- **Existing US-013 test compatibility:** The US-013 integration tests (`tests/us-013/fingerprint-receipt.test.ts`) assert exactly 12 `.receipt-row` elements via `receipt.querySelectorAll('.receipt-row')`. The plan adds the rerun button in a `.receipt-actions` container that is a sibling of `.receipt-rows`, not inside it. However, `renderFingerprintReceipt()` is being refactored — the function's return value will now include the actions container. The existing test queries `.receipt-row` elements which are inside `.receipt-rows`, so the selector still matches exactly 12 rows. **Verified: no regression risk** for US-013 tests, as `.receipt-actions` is a separate container and the row count is unchanged.

- **US-011 receipt component not modified:** The plan correctly does not modify `src/ui/receipt.ts` — it extends behavior in the fingerprint module's orchestrator (`fingerprint-receipt.ts`) by appending the actions container to the receipt element returned by `createReceipt()`. This preserves the receipt component's API contract.

- **`renderFingerprintReceipt()` signature change:** The function currently returns `HTMLElement` synchronously. The plan maintains this — the rerun callback is async but the initial render is synchronous. The returned element contains the button that triggers async behavior on click. No signature change needed. No breaking change to consumers.

- **Artificial delay implementation:** The ~300ms minimum delay should use `Promise.all([onRerun(), new Promise(r => setTimeout(r, 300))])` or similar to ensure the delay runs concurrently with the callback, not sequentially. This is an implementation detail, not an architectural concern.

- **Button error handling:** If the rerun callback (`collectSnapshot` + `formatSnapshotToRows`) throws, the button should still re-enable and restore its text. The plan's step 1 describes restore-after-callback behavior but does not explicitly mention error handling. The implementation should wrap the callback in try-finally to guarantee button state restoration. Low risk — `collectSnapshot()` and `formatSnapshotToRows()` are designed with per-field error handling and should not throw.

- **`receipt-rerun-btn--loading` class convention:** Uses BEM modifier syntax, consistent with the `osint-card` component's `data-confidence` attribute pattern and the project's general CSS class naming. Acceptable.

- **Note: `renderFingerprintReceipt` is not yet wired into the app bootstrap:** The function is exported from `src/modules/fingerprint/index.ts` but not called in `src/app/main.ts` or `src/layout/shell.ts`. The receipt exists as a standalone module. US-017's changes to the function are internal — they don't affect how or where the receipt is eventually mounted. This is not a risk but is worth noting for implementation context.

**Consistency with existing architecture:** Fully consistent. The rerun button factory follows the established `src/ui/` component pattern (pure function accepting typed input, returning `HTMLElement`). CSS additions to `src/styles/receipt.css` follow the shared component styling pattern. The fingerprint module's orchestrator correctly separates data collection (`collectSnapshot`) from rendering (`createReceipt`, DOM manipulation). The `aria-live` addition meets CLAUDE.md accessibility requirements. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Enables users to refresh signal data without page reload, directly fulfilling all four US-017 acceptance criteria
- **Positive:** Adds the second `src/ui/` component (`rerun-button.ts`), reinforcing the pure-function component pattern established by `receipt.ts`
- **Positive:** Reuses existing `.btn` and `.btn-secondary` shared styles, avoiding CSS duplication
- **Positive:** `aria-live="polite"` ensures screen readers announce content updates after rerun, meeting accessibility requirements
- **Positive:** Reduced-motion support for loading animation meets CLAUDE.md accessibility requirements
- **Positive:** Button placement outside `.receipt-rows` avoids focus loss and preserves existing test selectors
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** ~300ms artificial delay is a UX trade-off — improves perceived responsiveness but adds latency to an otherwise instant operation
- **Trade-off:** The rerun button factory is generic enough for reuse but is currently specific to the receipt context; if other modules need rerun functionality, the factory can be generalized later without architectural changes
