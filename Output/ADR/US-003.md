## ADR-003: Static Local-Only Architecture Policy

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-003

### Context

The project's core trust contract (CLAUDE.md) mandates no cookies, no analytics, no tracking, no backend profile creation, and all analysis running locally in-browser. Currently there are no automated guardrails to enforce these constraints ŌĆö a developer could inadvertently add `google-analytics`, `segment`, or a backend framework like `express` without any CI-time or build-time check catching it.

### Decision

Introduce a multi-layered architecture policy enforcement system:

1. **Policy definition** (`src/policies/architecture-policy.ts`) ŌĆö centralized banned dependency patterns and validation logic
2. **Build-time validation script** (`scripts/validate-architecture.ts`) ŌĆö checks package.json and Vite config against policy, wired into build via `npm run policy:check && vite build`
3. **Static analysis via ESLint** ŌĆö `no-restricted-imports` rule banning analytics/tracking modules; new `npm run lint` script (required by CLAUDE.md but previously missing)
4. **Test-time verification** ŌĆö unit tests for policy logic, integration tests asserting dist/ contains only static assets, and build-output scanning for tracking script references

### Architectural Impact Assessment

**Scope:** Medium ŌĆö adds new directories (`src/policies/`, `scripts/`, `tests/us-003/`), new devDependencies (eslint, @typescript-eslint/*, tsx), and modifies build pipeline.

**New dependencies:** ESLint + TypeScript parser/plugin (devDependencies only), tsx (devDependency for running validation script). These are standard tooling dependencies that don't affect the runtime bundle.

**Build pipeline change:** The `build` script gains a policy pre-check gate (`npm run policy:check && vite build`). This adds a few seconds to builds but prevents privacy-violating dependencies from shipping.

**Risks identified and mitigations:**
- ESLint devDependencies must be excluded from the policy checker's runtime dependency scan (the plan acknowledges this ŌĆö policy should check `dependencies` vs `devDependencies` separately)
- `no-restricted-imports` only catches static imports; build-output scanning (step 8) covers the dynamic import gap
- The banned list requires ongoing maintenance but is documented and extensible
- `tsx` adds a devDependency; acceptable for tooling scripts

**Consistency with existing architecture:** Aligns well ŌĆö project already uses Vite for static output, has Vitest for testing, and the plan fills gaps noted in CLAUDE.md (missing `npm run lint` and `npm run test` scripts). No runtime code changes; all additions are tooling/policy layer.

**No breaking changes** to existing code or tests. The US-001 build-output test pattern is consistent with the new static-deploy integration test approach.

### Consequences

- **Positive:** Automated enforcement of the project's trust contract; future dependency additions are gated; lint and test scripts now match CLAUDE.md requirements
- **Positive:** Multi-layered defense (dependency check + import lint + build output scan) provides defense-in-depth
- **Negative:** Adds ~4 devDependencies and build-time overhead; banned list needs periodic updates
- **Trade-off:** Custom ESLint rule for fetch/XHR external endpoint detection may be fragile ŌĆö build-output scanning is the more reliable backstop
