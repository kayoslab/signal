## ADR-008: Rendering Engine Signal Collection Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-008

### Context

The application needs to collect rendering engine signals (WebGL support, GPU renderer string, vendor string, WebGL version) so downstream modules (Fingerprint Receipt, Shadow Profile, Threat Model) can estimate device uniqueness and hardware traits. The `src/signals/` directory already contains two signal collectors — `locale-signals.ts` (US-006) and `device-signals.ts` (US-007) — establishing a clear pattern of typed interface + pure collector function + safe fallbacks + feature detection. US-008 is the third collector following these conventions.

### Decision

Add a rendering engine signal collector (`src/signals/rendering-signals.ts`) that:

1. Defines a `RenderingSignals` interface with fields for `webglSupported` (boolean | string), `renderer` (string), `vendor` (string), and `webglVersion` (string), following the established union-type fallback pattern from `device-signals.ts`
2. Implements internal helpers `tryWebGLContext()` and `extractRendererInfo(gl)` for WebGL context creation and debug extension querying — not exported, consistent with the private-helper pattern in `device-signals.ts`
3. Exports a pure function `collectRenderingSignals()` that orchestrates context creation, renderer extraction, and explicit context cleanup via `WEBGL_lose_context` extension
4. Returns `'unavailable'` fallback strings (using the `UNAVAILABLE` constant pattern from `device-signals.ts`) when WebGL or the debug extension is blocked, unsupported, or throws
5. Unit tests in `tests/us-008/` covering happy-path, fallback, and edge-case scenarios

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to the existing `src/signals/` directory and one new test directory (`tests/us-008/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. WebGL and Canvas are standard Web Platform APIs. Test mocking uses existing Vitest capabilities already configured in the project.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover the new files without modification.

**Risks identified and mitigations:**
- **`WEBGL_debug_renderer_info` deprecation:** Firefox has removed this extension, and other browsers may follow. The plan correctly treats renderer/vendor as optional — returning `'unavailable'` when the extension is null. This is the correct defensive approach and means the collector degrades gracefully as browsers tighten privacy controls.
- **WebGL context resource leaks:** Creating a WebGL context allocates GPU resources. The plan explicitly calls `loseContext()` via the `WEBGL_lose_context` extension for cleanup. If that extension is also unavailable, the context will be garbage-collected eventually, but the explicit cleanup is best practice. Risk is low — a single short-lived context for signal collection is negligible.
- **Canvas/WebGL mocking complexity in Vitest:** Vitest runs in Node (with happy-dom providing DOM globals), but happy-dom does not implement `HTMLCanvasElement.getContext('webgl')`. The test plan requires mocking `document.createElement` to return a stub canvas with a mock `getContext` method, which in turn returns mock GL objects. This is a deeper mock chain than the existing signal tests but follows the same `vi.stubGlobal()` / `Object.defineProperty()` patterns already used in US-006 and US-007 tests. Risk is low — the mock chain is well-defined and the plan covers all branches.
- **SecurityError in privacy-hardened browsers:** Some browsers throw `SecurityError` when creating canvas or WebGL contexts (e.g., via resistFingerprinting settings). The plan wraps context creation in try-catch, returning full-unavailable results on any exception. This is consistent with the storage detection pattern in `device-signals.ts`.
- **Empty renderer strings:** Some browsers return empty strings from `getParameter()` even when the debug extension is available. The plan should treat empty strings as equivalent to unavailable — a minor implementation detail, not an architectural concern.

**Consistency with existing architecture:** Fully consistent. The plan follows the exact same pattern established by US-006 and US-007: typed interface with union-type fallbacks, pure exported collector function, internal helpers, feature detection before every API access, try-catch around potentially throwing APIs, and `UNAVAILABLE` constant for fallback values. Files land in `src/signals/` per CLAUDE.md conventions. Tests land in `tests/us-008/` following the per-ticket test directory convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Adds the third signal collector, further reinforcing the established pattern and providing GPU/rendering data for uniqueness estimation in downstream modules
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** Explicit WebGL context cleanup prevents GPU resource leaks during signal collection
- **Positive:** Graceful degradation when `WEBGL_debug_renderer_info` is deprecated/removed means the module will continue functioning as browsers evolve their privacy stance
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** None significant — this is a minimal, additive change with no architectural risk
