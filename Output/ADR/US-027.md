## ADR-027: Threat Model Rules Engine

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-027

### Context

The application needs a deterministic rules engine that maps browser signal findings to user-relevant risks for the Threat Model dashboard module (CLAUDE.md Required Module #6). The `src/modules/threat-model/` directory does not yet exist. The plan depends on four upstream modules: `src/signals/snapshot.ts` (US-009) providing `SignalSnapshot`, `src/permissions/permissions-adapter.ts` (US-021) providing `PermissionCheckResult[]`, `src/scoring/entropy-score.ts` (US-014) providing `EntropyResult`, `src/scoring/permission-debt-score.ts` (US-022) providing `PermissionDebtResult`, and `src/modules/shadow-profile/inference-schema.ts` (US-024/US-025) providing `InferenceStatement[]`. All five upstream interfaces are implemented and stable.

### Decision

Add a threat model rules engine with:

1. **Schema definitions** (`src/modules/threat-model/threat-schema.ts`) â€” `ThreatRule` interface (id, title, category, evaluate function, explanation), `ThreatFinding` interface (ruleId, title, severity `'Low' | 'Medium' | 'High'`, description, evidence, userImpact, category), and `ThreatInput` interface composing `SignalSnapshot`, `PermissionCheckResult[]`, `EntropyResult`, `PermissionDebtResult`, and `InferenceStatement[]`
2. **Rules engine** (`src/modules/threat-model/threat-rules.ts`) â€” pure function `evaluateThreatRules(input: ThreatInput): ThreatFinding[]` executing four category rules (identity-exposure, social-engineering, permission-abuse, shoulder-surfing) and collecting non-null findings
3. **Four individual rule functions** â€” each a small pure function returning `ThreatFinding | null`
4. **Explainability helper** â€” `getRuleExplanations(): Record<string, string>` for self-documenting rules
5. **Unit tests** (`tests/us-027/threat-rules.test.ts`) covering each rule in isolation with factory helpers

### Architectural Impact Assessment

**Scope:** Small â€” adds a new directory (`src/modules/threat-model/`) with two source files (schema + rules engine) and one new test directory (`tests/us-027/`). No new dependencies. No build pipeline changes. No configuration changes. No modifications to existing files.

**New dependencies:** None. The module imports types from existing `src/signals/snapshot.ts`, `src/permissions/permissions-adapter.ts`, `src/scoring/entropy-score.ts`, `src/scoring/permission-debt-score.ts`, and `src/modules/shadow-profile/inference-schema.ts`. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new files without modification.

**No bootstrap/main.ts changes:** This ticket is purely data-layer â€” it adds rule evaluation logic but does not mount anything to the DOM or modify `src/app/main.ts`. A future rendering ticket will consume `evaluateThreatRules()` to build the dashboard module. This separation is correct per CLAUDE.md's mandate to separate data collection from rendering and to "never mix DOM logic into scoring modules."

**Risks identified and mitigations:**

- **Threshold tuning is subjective (entropy â‰Ą 70, debt â‰Ą 40):** The entropy score range is 0â€“100 (from `EntropyResult.score`). A threshold of 70 for identity-exposure means roughly 70% of weighted signals are present and distinguishing. The `ENTROPY_WEIGHTS` sum to 84 (`maxPossible`), so a score of 70 corresponds to having most high-weight signals (renderer=15, languages=12, screenResolution=10, timezone=10) available. This is a reasonable heuristic boundary. Similarly, permission debt â‰Ą 40 means at least one high-risk permission (camera/microphone/geolocation at weight 10 each contribute ~14% each of a maxPossible ~72). Both thresholds are implemented as constants, trivially adjustable without architectural changes. Per CLAUDE.md: "all heuristics must be explainable" â€” the `explanation` field on each rule satisfies this requirement.

- **Shoulder-surfing rule heuristics may misclassify:** The rule triggers when `screenWidth >= 1920`, `devicePixelRatio <= 1`, and `touchSupport === false`. These conditions must handle the union types in `DeviceCapabilitySignals`: `screenWidth` is `number | string`, `devicePixelRatio` is `number | string`, `touchSupport` is `boolean | string`. The implementation must perform type narrowing (`typeof snapshot.device.screenWidth === 'number'`) before numeric comparisons. If any value is a string sentinel (`'unavailable'`), the condition should fail and the rule should return `null`. This is the same type-narrowing pattern used in `src/modules/shadow-profile/inference-rules.ts` (US-025) and `src/scoring/entropy-score.ts` (US-014). Severity is always `'Low'` for this rule, which is appropriately conservative for a physical-proximity heuristic.

- **`ThreatInput` is a composite of five upstream interfaces:** This creates the widest input contract of any module in the codebase. The `ThreatInput` interface depends on: `SignalSnapshot` (stable since US-009), `PermissionCheckResult[]` (stable since US-021), `EntropyResult` (stable since US-014), `PermissionDebtResult` (stable since US-022), and `InferenceStatement[]` (stable since US-024/US-025). All five interfaces are implemented, tested, and consumed by at least one other module. The risk of interface breakage is low. If any upstream interface is extended with optional fields, `ThreatInput` is unaffected (TypeScript structural typing allows additive changes). The wider concern is that the rules engine caller must orchestrate all five data sources â€” this is a rendering-layer concern for the future dashboard ticket, not an architectural risk for US-027's data-layer scope.

- **Dependency on US-025 `InferenceStatement` shape:** The social-engineering rule consumes `InferenceStatement[]` to count inferences with medium+ confidence. The `InferenceStatement` interface (`statement`, `evidence`, `confidence`, `marker`) is minimal and enforced by the `createInference()` factory. The rule only reads the `confidence` field (type `ConfidenceLabel = 'low' | 'medium' | 'high'`), so it is resilient to any future additions to the interface. The plan correctly identifies this dependency as a risk but the interface is stable and the consumption is minimal.

- **Content wording must avoid fear language:** Per CLAUDE.md content rules ("never exaggerate risk," "avoid fear language," "use concise executive-grade wording"), the `description` and `userImpact` strings in each `ThreatFinding` must use measured, factual language. For example, "Your browser signals are sufficiently unique to enable cross-site tracking" rather than "Hackers can track your every move." This is a content concern validated at the test level (regex scan for certainty/fear words), consistent with US-025's approach. Not an architectural risk.

- **Severity labels must be exactly `'Low' | 'Medium' | 'High'`:** The acceptance criteria explicitly require these three labels. TypeScript's literal type enforcement guarantees compile-time correctness. The plan's test recommendations correctly include a test verifying severity values are strictly from this set. This directly satisfies CLAUDE.md Required Module #6: "Use Low / Medium / High severity badges."

- **`getRuleExplanations()` is a static helper:** It returns a record mapping rule IDs to explanation strings. This is a read-only helper with no state, no side effects, and no upstream dependencies. It exists solely for UI consumption and test verification of explainability. Zero risk.

- **No barrel export (`index.ts`) mentioned in plan:** The plan does not explicitly list an `index.ts` barrel export, unlike other module directories (`fingerprint/index.ts`, `shadow-profile/index.ts`, `permission-debt/index.ts`). The implementation should add one for consistency with the established module pattern. This is a minor omission, not an architectural concern.

- **Test factory helpers for `ThreatInput`:** The `ThreatInput` composite requires factory helpers that build all five sub-objects. The test should use the existing `makeFullSnapshot()` pattern (from `tests/us-020/`) for `SignalSnapshot`, and create minimal factory functions for the other four types. Each factory should produce valid objects with sensible defaults that can be selectively overridden per test case. This is standard test infrastructure and follows the established convention.

**Consistency with existing architecture:** Fully consistent. The module follows established conventions: per-module subdirectory in `src/modules/`, typed interfaces for data contracts, pure functions with no side effects, separation of data/scoring logic from DOM rendering. The rule-evaluation pattern (input â†’ array of findings, each rule returning `T | null`, aggregator filtering nulls) mirrors the `applyInferenceRules()` pattern from `src/modules/shadow-profile/inference-rules.ts` (US-025). The `ThreatFinding` result interface with `severity` and `evidence` fields is structurally analogous to `InferenceStatement` with `confidence` and `evidence`. The `explanation` field on rules satisfies CLAUDE.md's "all heuristics must be explainable" coding rule. Files land in `src/modules/threat-model/` per CLAUDE.md conventions (threat model is Required Module #6). Tests land in `tests/us-027/` following the per-ticket convention. No runtime dependencies added. All computation is client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. This ticket is purely additive â€” no existing files are modified.

### Consequences

- **Positive:** Delivers deterministic threat-to-risk mappings for all four required categories (identity-exposure, social-engineering, permission-abuse, shoulder-surfing), directly fulfilling all five US-027 acceptance criteria
- **Positive:** Pure-function rule design makes each rule independently testable with deterministic inputs/outputs, consistent with the inference rules pattern from US-025
- **Positive:** `explanation` field on every rule satisfies CLAUDE.md's "all heuristics must be explainable" coding rule and the ticket's "rules must be explainable" note
- **Positive:** Severity labels strictly typed as `'Low' | 'Medium' | 'High'` satisfy acceptance criteria and CLAUDE.md Required Module #6 specification
- **Positive:** Purely additive â€” zero risk to existing modules, tests, or build pipeline
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All computation remains client-side; no trust rule violations
- **Negative:** `ThreatInput` composite type creates the widest input contract in the codebase (five upstream interfaces) â€” mitigated by all interfaces being stable and the consumption being read-only
- **Negative:** Threshold values (entropy â‰Ą 70/85, debt â‰Ą 40, screen â‰Ą 1920, DPR â‰¤ 1) are heuristic approximations that may need tuning after real-world testing â€” mitigated by named constants and the `explanation` field documenting rationale
- **Trade-off:** Four rules in a single file is appropriate for current scope; if rules grow beyond ~8, extraction into individual files or a registry pattern would improve maintainability, but premature at this stage
