## ADR-012: Fingerprint Receipt Data Formatting Module

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-012

### Context

The application needs to display core browser fingerprint signals (timezone, languages, platform, touch support, CPU threads, Do Not Track) in a human-readable receipt format. The `src/signals/` barrel export (US-009) provides `collectSnapshot()` returning a frozen `SignalSnapshot` with `locale`, `device`, and `rendering` sub-objects, plus `MISSING` sentinel constants (`unknown`, `unavailable`). The `src/ui/receipt.ts` (US-011) provides a reusable `createReceipt(title, rows)` component accepting `ReceiptRow[]` (label/value string pairs). The `src/modules/` directory exists but is empty (`.gitkeep` only). The `src/app/main.ts` currently calls `renderShell(app)` synchronously with no signal consumption.

### Decision

Add a fingerprint receipt formatting module with:

1. **Formatting function** (`src/modules/fingerprint-receipt.ts`) — pure function `formatSnapshotToReceiptRows(snapshot: SignalSnapshot): ReceiptRow[]` that extracts six fields from the snapshot and transforms them into human-readable label/value pairs: timezone (passthrough), languages (comma-separated), platform (passthrough), touch support ('Supported'/'Not supported'), CPU threads (e.g. '8 threads' or 'Unavailable'), Do Not Track ('Enabled'/'Disabled'/'Not set')
2. **Sentinel value handling** — maps `MISSING.unknown` and `MISSING.unavailable` to user-friendly display strings like 'Unavailable' rather than exposing raw sentinel values
3. **Bootstrap wiring** (`src/app/main.ts`) — call `collectSnapshot()`, pass through `formatSnapshotToReceiptRows()`, then render via `createReceipt('Fingerprint Receipt', rows)` into the dashboard container
4. **Tests** (`tests/us-012/fingerprint-receipt.test.ts`) — field formatting, missing value fallbacks, row count verification, DOM rendering integration

### Architectural Impact Assessment

**Scope:** Very small — adds one source file to the existing `src/modules/` directory, modifies one existing file (`src/app/main.ts`), and adds one new test directory (`tests/us-012/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/signals/` and `src/ui/receipt.ts` modules. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Bootstrap flow change:** `src/app/main.ts` currently calls `renderShell(app)` only. The plan adds `collectSnapshot()` and receipt rendering after the shell. This is a minor additive change — the shell still renders first, and the receipt is appended into it. If `collectSnapshot()` or formatting fails, it should not prevent the shell from rendering (the plan should wrap the receipt wiring in try-catch as a defensive measure, though the snapshot's per-collector error handling makes failures unlikely).

**Risks identified and mitigations:**
- **First consumer of `SignalSnapshot` in the UI layer:** This module establishes the pattern for how downstream modules consume snapshot data. The pure-function approach (`formatSnapshotToReceiptRows`) correctly separates data transformation from rendering, consistent with CLAUDE.md's mandate to separate data collection from rendering. Future modules (OSINT Cards, Shadow Profile, Threat Model) should follow this same pattern.
- **Human-readable formatting choices:** Labels like 'Not set' vs 'Unknown' for Do Not Track, or 'Unavailable' for missing CPU threads, must align with CLAUDE.md content rules (no exaggeration, concise wording). The plan's choices ('Enabled'/'Disabled'/'Not set', 'Unavailable') are appropriate — they are factual and concise without overstating risk.
- **Hardcoded 6-row output:** The function returns exactly 6 `ReceiptRow` objects mapping to the 6 acceptance criteria fields. This is intentionally rigid — the acceptance criteria are explicit about which fields to show. Future tickets can extend the row set if needed.
- **`MISSING` sentinel matching:** The formatter must check field values against both `MISSING.unknown` and `MISSING.unavailable` strings. Since these are exported constants from `src/signals/snapshot.ts`, the comparison is type-safe and refactor-proof.
- **Placement in `src/modules/` vs `src/modules/fingerprint-receipt/`:** The plan places the file directly in `src/modules/` rather than a subdirectory. US-010 (intro sequence) used a subdirectory (`src/modules/intro/`) because it had multiple files (component + CSS + counter). A single formatting file without co-located CSS is appropriately placed directly in `src/modules/`. If a future ticket adds fingerprint-receipt-specific styles, it can be refactored into a subdirectory then.

**Consistency with existing architecture:** Fully consistent. The module bridges two existing layers (`src/signals/` → `src/ui/`) with a pure formatting function, which is exactly the separation of concerns CLAUDE.md mandates. The `ReceiptRow` interface from `src/ui/receipt.ts` is the contract — the formatter produces rows, the receipt component renders them. This is a clean, testable data pipeline. Files land in appropriate directories per CLAUDE.md conventions. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The `main.ts` modification is additive — existing shell rendering is preserved.

### Consequences

- **Positive:** Delivers the first visible data in the dashboard — users see real browser signals immediately after the shell renders
- **Positive:** Establishes a clean pattern for formatting snapshot data into receipt rows that future modules can follow
- **Positive:** Pure function design makes formatting logic trivially testable without DOM dependencies
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** None significant — this is a minimal, straightforward data formatting layer with no architectural risk
- **Trade-off:** Placing the file directly in `src/modules/` (not a subdirectory) is appropriate for a single-file module but may need restructuring if the fingerprint receipt gains its own styles or sub-components in future tickets
