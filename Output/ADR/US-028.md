## ADR-028: Render Threat Model Dashboard Module

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-028

### Context

The application needs a Threat Model dashboard module (CLAUDE.md Required Module #6) that translates browser signal findings into user-relevant risks with severity badges. The `src/modules/threat-model/` directory already contains `threat-schema.ts` (US-027) with `ThreatFinding`, `ThreatInput`, `ThreatRule`, `Severity`, and `ThreatCategory` type definitions, plus `threat-rules.ts` (US-027) with `evaluateThreatRules(input: ThreatInput): ThreatFinding[]` evaluating four category rules (identity-exposure, social-engineering, permission-abuse, shoulder-surfing). The `src/app/main.ts` currently mounts three modules into `#dashboard`: `renderZeroClickOsintModule()`, `renderPermissionDebtModule()`, and `renderShadowProfileModule()`. The `src/ui/shadow-profile-card.ts` establishes the card component pattern: `createShadowProfileCard(inference)` returning an article element, plus `createShadowProfileCardList(cards)` for grid container.

### Decision

Add a Threat Model rendering module with:

1. **Threat card UI component** (`src/ui/threat-model-card.ts`) â€” `createThreatCard(finding: ThreatFinding): HTMLElement` rendering an article element with severity badge (`Low`/`Medium`/`High` with modifier class), title heading, short description, evidence list, userImpact blurb, and category label. `createThreatCardList(cards: HTMLElement[]): HTMLElement` wrapping cards in a responsive grid container. Follows the `createShadowProfileCard`/`createShadowProfileCardList` pattern.
2. **Module styles** (`src/modules/threat-model/threat-model.css`) â€” section wrapper `.threat-model`, card styles `.threat-model-card`, severity badge modifiers `--low`/`--medium`/`--high` using existing semantic tokens (`--color-success`, `--color-warning`, `--color-danger`), responsive grid layout (1-col mobile, 2-col at `--breakpoint-md`), hover shadow transition, and `prefers-reduced-motion` support.
3. **Module render function** (`src/modules/threat-model/threat-model-module.ts`) â€” async `renderThreatModelModule(): Promise<HTMLElement>` collecting snapshot, permissions, entropy score, permission debt score, and inferences, assembling `ThreatInput`, calling `evaluateThreatRules()`, mapping findings to cards, returning section element with h2 heading. Handles empty state gracefully.
4. **Barrel export** (`src/modules/threat-model/index.ts`) â€” re-export `renderThreatModelModule`.
5. **Bootstrap wiring** (`src/app/main.ts`) â€” mount `renderThreatModelModule()` into `#dashboard` after shadow profile rendering.
6. **CSS import** â€” add `threat-model.css` import to `src/styles/global.css`, following the existing pattern for `shadow-profile.css` and `osint-card.css`.
7. **Unit tests** (`tests/us-028/threat-model-card.test.ts`) â€” card DOM structure, severity modifier classes for all three levels, card list container children.
8. **Integration tests** (`tests/us-028/threat-model-module.test.ts`) â€” module returns section element, renders risk cards, shows at least 4 categories with mocked high-entropy/granted-permissions input, handles empty findings gracefully.

### Architectural Impact Assessment

**Scope:** Small â€” adds two new source files (`src/ui/threat-model-card.ts`, `src/modules/threat-model/threat-model-module.ts`), one new CSS file (`src/modules/threat-model/threat-model.css`), one barrel file (`src/modules/threat-model/index.ts`), modifies two existing files (`src/app/main.ts`, `src/styles/global.css`), and adds one new test directory (`tests/us-028/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The module imports from existing `src/signals/`, `src/permissions/`, `src/scoring/`, `src/modules/shadow-profile/`, and the existing `src/modules/threat-model/threat-rules.ts` (US-027). No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Bootstrap flow change:** `src/app/main.ts` currently mounts `renderZeroClickOsintModule()` (synchronous), `renderPermissionDebtModule()` (async), and `renderShadowProfileModule()` (async) into `#dashboard`. The plan adds `renderThreatModelModule()` (async â€” due to `checkPermissions()` returning a Promise) as the fourth module. This follows the established async mounting pattern. The insertion order (zero-click â†’ permission-debt â†’ shadow-profile â†’ threat-model) aligns with CLAUDE.md Required Modules ordering (#3 â†’ #4 â†’ #5 â†’ #6). If `renderThreatModelModule()` throws, it should not prevent prior modules from rendering â€” the implementation should wrap the mount in try-catch, consistent with defensive patterns used by existing module mounts.

**Risks identified and mitigations:**

- **`ThreatInput` assembly requires orchestrating five data sources:** The render function must call `collectSnapshot()`, `checkPermissions()`, `calculateEntropyScore(snapshot)`, `calculatePermissionDebtScore(permissions)`, and `applyInferenceRules({ snapshot, permissions })` to build the `ThreatInput`. This is the widest data orchestration of any module in the codebase. However, each upstream function is well-tested and designed with per-field error handling. The `checkPermissions()` call is the only async operation and must be wrapped in try-catch â€” the same pattern used in `shadow-profile-module.ts`. If permissions fail, the module should still function with an empty permissions array, allowing entropy-based and snapshot-based rules to still fire.

- **`applyInferenceRules` import path from sibling module:** The plan imports `applyInferenceRules` from `src/modules/shadow-profile/inference-rules.ts`. This is a cross-module dependency within `src/modules/` â€” the threat-model module depends on the shadow-profile module's inference engine. This is architecturally acceptable because the inference rules are a pure data-layer function with no DOM coupling, and the dependency direction is acyclic (threat-model â†’ shadow-profile, never the reverse). The import path is straightforward and the barrel export at `src/modules/shadow-profile/index.ts` already re-exports `applyInferenceRules`.

- **Shoulder-surfing rule depends on `window.screen` dimensions in jsdom:** The `evaluateShoulderSurfing` rule checks `snapshot.device.screenWidth >= 1920` and `devicePixelRatio <= 1`. In jsdom/happy-dom test environments, these values must be mocked via a controlled `SignalSnapshot` fixture. The integration test should mock `collectSnapshot` at the module level to return a deterministic snapshot, consistent with `tests/us-020/` and `tests/us-026/`. This is standard practice and does not require any special test environment configuration.

- **`checkPermissions()` rejection could fail the entire module:** If `navigator.permissions.query()` throws in the test environment or a restricted browser context, the async call rejects. The implementation must use the same try-catch pattern from `shadow-profile-module.ts`: catch the rejection, fall back to an empty `PermissionCheckResult[]`, and continue with the remaining data sources. This ensures the module always renders â€” at minimum showing entropy-based findings.

- **CSS import strategy â€” module directory vs global.css:** The plan places `threat-model.css` in `src/modules/threat-model/` (co-located with the module) and imports it via `src/styles/global.css`. This follows the exact pattern used by `shadow-profile.css` â€” co-located in `src/modules/shadow-profile/` and imported via `global.css`. While this creates a cross-directory import in `global.css`, the pattern is already established and consistent.

- **Severity badge color tokens verified:** `--color-success` (#22c55e for Low), `--color-warning` (#eab308 for Medium), and `--color-danger` (#ef4444 for High) all exist in `src/styles/tokens.css`. No token creation needed. The color mapping (Low=success/green, Medium=warning/yellow, High=danger/red) follows intuitive severity escalation, unlike the inverted mapping in permission-debt (where granted=danger).

- **At least four risk categories acceptance criterion:** The four rules in `threat-rules.ts` map to four distinct categories: `identity-exposure`, `social-engineering`, `permission-abuse`, `shoulder-surfing`. To trigger all four simultaneously in the integration test, the mocked input must have: entropy score â‰Ą 70 (identity-exposure), â‰Ą 2 medium+ confidence inferences (social-engineering), at least one high-risk granted permission (permission-abuse), and screen â‰Ą 1920px with DPR â‰¤ 1 and no touch (shoulder-surfing). This is achievable with a carefully constructed `ThreatInput` fixture.

- **Empty state handling:** If no rules fire (e.g., all signals are missing sentinels, no permissions granted, low entropy), `evaluateThreatRules()` returns an empty array. The plan requires graceful empty-state rendering with an informative message. This is consistent with the empty-state handling in `shadow-profile-module.ts`.

- **Duplicate `collectSnapshot()` call:** Also called by intro, zero-click-osint, and shadow-profile modules. Each call creates a new frozen snapshot. The signal collectors are lightweight and idempotent. Negligible performance impact.

- **Content wording in threat findings:** The `description` and `userImpact` strings are defined in `threat-rules.ts` (US-027), not in this rendering ticket. This module renders them as-is. The US-027 ADR already verified that content follows CLAUDE.md rules (no exaggeration, no fear language, concise executive-grade wording).

- **Existing test compatibility:** The threat-model module is new and no existing tests reference it. The `main.ts` modification is additive â€” existing modules continue to render in their established order. No test regressions.

**Consistency with existing architecture:** Fully consistent. The module follows the established subdirectory pattern (`src/modules/threat-model/` with barrel export), consistent with `intro/`, `fingerprint/`, `permission-debt/`, `zero-click-osint/`, and `shadow-profile/`. The render function returns `Promise<HTMLElement>`, matching the async pattern from `renderPermissionDebtModule` and `renderShadowProfileModule`. The threat card component follows the exact `src/ui/` component pattern (pure function accepting typed input, returning `HTMLElement`, using `document.createElement` with class assignment) established by `shadow-profile-card.ts` and `osint-card.ts`. CSS uses existing design tokens and follows the module-specific co-located pattern. The module composes existing upstream layers (`src/signals/` + `src/permissions/` + `src/scoring/` + `src/modules/shadow-profile/inference-rules` + `src/modules/threat-model/threat-rules` â†’ `src/ui/threat-model-card`) in the expected dependency direction. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment. The `main.ts` modification is additive â€” existing intro, shell, zero-click-osint, permission-debt, and shadow-profile rendering are preserved.

### Consequences

- **Positive:** Delivers CLAUDE.md Required Module #6, translating technical findings into user-relevant risks with Low/Medium/High severity badges, directly fulfilling all four acceptance criteria
- **Positive:** Follows established module and UI component patterns, reinforcing architectural consistency across `src/modules/`, `src/ui/`, and `src/styles/`
- **Positive:** Composes five existing upstream layers cleanly, validating the layered architecture design at its widest integration point
- **Positive:** The `createThreatCard` component is purpose-built for the `ThreatFinding` interface, ensuring severity badges, evidence, and user impact are consistently displayed
- **Positive:** Reduced-motion CSS support meets CLAUDE.md accessibility requirements
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations
- **Negative:** `ThreatInput` assembly in the render function creates the widest data orchestration in the codebase (five upstream calls) â€” mitigated by all upstream functions being stable, tested, and designed with error resilience
- **Negative:** `collectSnapshot()` is called redundantly (also called by intro, zero-click-osint, and shadow-profile modules) â€” negligible performance impact; can be optimized via snapshot passing in a future ticket
- **Negative:** Cross-module dependency on `applyInferenceRules` from shadow-profile creates coupling between modules #5 and #6 â€” acceptable because the dependency is on a pure data function, is acyclic, and the inference rules are a shared data concern rather than a UI concern
- **Trade-off:** Co-located CSS in `src/modules/threat-model/` imported via `global.css` follows the shadow-profile precedent but creates a cross-directory import; functionally equivalent to placing in `src/styles/` but keeps module assets co-located
