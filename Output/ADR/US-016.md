## ADR-016: Add Receipt Copy Action

**Status:** Accepted  
**Date:** 2026-04-24  
**Ticket:** US-016

### Context

The application needs a copy button on the Fingerprint Receipt so users can copy receipt text to the clipboard for sharing or documentation. The `src/modules/fingerprint/fingerprint-receipt.ts` (US-012, US-013, US-015, US-017) renders a receipt with 15 signal+posture rows, a rerun button in a `.receipt-actions` container, and `aria-live="polite"` on the rows container. The `src/ui/rerun-button.ts` (US-017) establishes the button factory pattern: pure function accepting a callback, returning `HTMLButtonElement`, with double-click guard, disabled/text/class state management, and try/catch/finally for resilience. The `src/styles/receipt.css` already defines `.receipt-actions` with flexbox layout and gap spacing. The `src/ui/receipt.ts` (US-011) exports the `ReceiptRow` interface (`{ label: string; value: string }`).

### Decision

Add a receipt copy action with:

1. **Copy button factory** (`src/ui/copy-button.ts`) â€” `createCopyButton(getReceiptText: () => string): HTMLButtonElement` following the `createRerunButton` pattern: `btn btn-secondary receipt-copy-btn` classes, `running` flag double-click guard, async handler with disabled/text/class toggles, try/catch/finally for resilience
2. **Clipboard interaction** â€” check `navigator.clipboard?.writeText` availability; if available, call `writeText(text)` and show success feedback ('Copied!' + `receipt-copy-btn--success` class); if unavailable or rejected, show failure feedback ('Copy unavailable' + `receipt-copy-btn--fail` class); revert after 2 seconds via `setTimeout`
3. **Accessibility** â€” `aria-live="assertive"` on a visually-hidden status span adjacent to the button for screen reader copy announcements
4. **CSS states** (`src/styles/receipt.css`) â€” `.receipt-copy-btn--success` and `.receipt-copy-btn--fail` with subtle color shifts using existing semantic design tokens (`--color-success` for success, `--color-text-muted` for fail), with `prefers-reduced-motion` support
5. **Text formatter** (`src/modules/fingerprint/format-receipt-text.ts`) â€” `formatReceiptText(rows: ReceiptRow[]): string` joining rows into a plain-text receipt block with header line ('â€” Fingerprint Receipt â€”') and formatted label-value pairs
6. **Receipt wiring** (`src/modules/fingerprint/fingerprint-receipt.ts`) â€” import `createCopyButton`, call with a closure invoking `formatReceiptText` on current rows, prepend to `.receipt-actions` container before the rerun button
7. **Unit tests** (`tests/us-016/copy-button.test.ts`, `tests/us-016/format-receipt-text.test.ts`) â€” button rendering, clipboard success/failure/unavailable paths, double-click guard, feedback revert, text formatting with sample rows, empty rows, special characters

### Architectural Impact Assessment

**Scope:** Very small â€” adds two new source files (`src/ui/copy-button.ts`, `src/modules/fingerprint/format-receipt-text.ts`), modifies two existing files (`src/modules/fingerprint/fingerprint-receipt.ts`, `src/styles/receipt.css`), and adds one new test directory (`tests/us-016/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. The Clipboard API is a standard Web Platform API. The button factory uses standard DOM APIs available in Vitest's happy-dom environment. No runtime or dev dependencies are added.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all new and modified files without modification.

**Risks identified and mitigations:**

- **Clipboard API requires secure context (HTTPS or localhost):** `navigator.clipboard` is undefined in non-secure contexts. The availability check (`navigator.clipboard?.writeText`) handles this gracefully â€” the button shows 'Copy unavailable' feedback rather than throwing. Directly satisfies the "failure feedback shown when unavailable" acceptance criterion.

- **Clipboard permissions may be denied by browser policy:** Some browsers require user gesture context for `clipboard.writeText()`, and enterprise policies may block clipboard access. The plan wraps `writeText` in try/catch, catching `DOMException` (permission denied) and showing failure feedback. Consistent with error handling in `createRerunButton`.

- **Timer-based feedback revert (setTimeout 2s) and component lifecycle:** If the receipt is removed from the DOM before the timeout fires, the callback executes on a detached element. In a static SPA where the receipt persists for the page lifetime, this is a non-issue. No cleanup mechanism needed for the current architecture.

- **Receipt rows must stay synchronized between rendering and text formatting:** The `formatReceiptText` function receives `ReceiptRow[]` â€” the same array used to build the DOM. The closure must capture rows dynamically (reading from a mutable `currentRows` reference) rather than closing over the initial array, otherwise the copy button will copy stale data after a rerun. Minor state management concern, not architectural risk.

- **Existing test compatibility:** The copy button is added to `.receipt-actions` (a sibling of `.receipt-rows`), not inside `.receipt-rows`. US-013 and US-015 tests query `.receipt-row` elements inside `.receipt-rows`, so the copy button does not affect row count assertions. US-017 tests assert the rerun button exists in `.receipt-actions` â€” adding a sibling copy button does not affect these assertions. **No test regressions.**

- **`aria-live="assertive"` for copy status vs `aria-live="polite"` on receipt rows:** The receipt rows container uses `polite` for rerun content updates. The copy status span uses `assertive` because clipboard feedback is a direct response to user action. Using `assertive` on a visually-hidden span avoids disrupting the button's accessible name. Correct pattern.

- **CSS token availability:** `--color-success` (#22c55e) and `--color-text-muted` exist in `src/styles/tokens.css`. No new token creation needed.

**Consistency with existing architecture:** Fully consistent. The copy button factory follows the exact pattern established by `createRerunButton`: pure function accepting typed input, returning `HTMLButtonElement`, with state management via class toggles and disabled property. CSS additions to `src/styles/receipt.css` follow the existing pattern of receipt-specific styles with BEM modifier classes. The text formatter follows CLAUDE.md's mandate to separate data logic from rendering. The wiring in `fingerprint-receipt.ts` follows the existing pattern of creating buttons via factory functions and appending to `.receipt-actions`. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All data remains client-side per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Delivers clipboard copy functionality for the fingerprint receipt, directly fulfilling all four US-016 acceptance criteria (button visible, text copied, success feedback, failure feedback)
- **Positive:** Follows the established button factory pattern from `createRerunButton`, maintaining architectural consistency in `src/ui/`
- **Positive:** Pure `formatReceiptText` function is independently testable and reusable for future text-based sharing features
- **Positive:** `aria-live="assertive"` on status span provides immediate screen reader feedback for clipboard operations, meeting CLAUDE.md accessibility requirements
- **Positive:** Graceful degradation when Clipboard API is unavailable (non-secure context, browser policy) ensures the feature never throws unhandled errors
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** All data remains client-side; no trust rule violations â€” clipboard write is a local browser operation
- **Negative:** Timer-based feedback revert (2s setTimeout) has no cleanup on component removal â€” acceptable for static SPA architecture where the receipt persists for page lifetime
- **Negative:** Copy button closure must reference fresh rows after rerun â€” requires mutable `currentRows` variable in `renderFingerprintReceipt` scope, adding minor state management complexity
- **Trade-off:** `aria-live="assertive"` may interrupt screen reader users mid-task if they trigger copy accidentally â€” mitigated by the double-click guard and the feedback being a brief, informative message
