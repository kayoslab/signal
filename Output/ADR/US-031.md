## ADR-031: Persistent Privacy Footer Trust Statements

**Status:** Accepted  
**Date:** 2026-04-23  
**Ticket:** US-031

### Context

The application requires explicit trust messaging visible at all times to reinforce the project's core privacy guarantees (CLAUDE.md Non-Negotiable Trust Rules). The existing footer in `src/layout/shell.ts` displays a single line: "All analysis runs locally. No data leaves your browser." This needs to be expanded to cover all four trust statements required by the acceptance criteria: no cookies, no analytics, no backend profiling, and all analysis runs locally.

### Decision

Enhance the existing footer with structured trust statements:

1. **Footer markup update** (`src/layout/shell.ts`) — replace the single `textContent` string with structured HTML elements (list or span series) containing all four trust statements, each individually identifiable via semantic markup or data attributes for testability
2. **Footer styling update** (`src/layout/shell.css`) — style trust statements as an inline horizontal list with separators using flexbox, existing design tokens, and responsive wrapping for mobile viewports
3. **Unit tests** (`tests/us-031/privacy-footer.test.ts`) — verify all four trust statements are present, footer uses semantic `<footer>` tag, and footer is persistently rendered

### Architectural Impact Assessment

**Scope:** Very small — modifies two existing files (`src/layout/shell.ts`, `src/layout/shell.css`), adds one new test directory (`tests/us-031/`). No new dependencies. No build pipeline changes. No configuration changes.

**New dependencies:** None. All changes use existing DOM APIs and CSS features already in use throughout the project.

**Build pipeline change:** None. Existing `npm run lint`, `npm run typecheck`, and `npm run test` scripts cover all modified and new files without modification.

**Risks identified and mitigations:**
- **Existing US-004 test compatibility:** Verified that US-004 tests (`tests/us-004/responsive-shell.test.ts`) assert only on footer existence, semantic tag name, and DOM order — they do not assert on footer text content. Therefore, changing the footer content carries zero regression risk for US-004.
- **Existing US-005 test compatibility:** US-005 tests verify theme tokens and shared component styles. The footer styling changes use existing design tokens (`--color-text-muted`, `--font-size-xs`, `--spacing-sm`, `--spacing-md`) without modifying them. Zero regression risk.
- **Footer wrapping on narrow viewports (< 320px):** Four trust statements in an inline list could look cluttered on very narrow screens. Flexbox `flex-wrap: wrap` with appropriate gap spacing mitigates this. The plan correctly identifies this as a risk to verify during manual testing.
- **Shield/lock icon addition:** Optional CSS pseudo-element or inline SVG for credibility is a minor visual enhancement. If implemented as a CSS pseudo-element, it adds no DOM complexity. If implemented as inline SVG, it adds minimal markup. Either approach is consistent with the project's lightweight styling philosophy.

**Consistency with existing architecture:** Fully consistent. The changes modify existing files in `src/layout/` per CLAUDE.md conventions. The footer remains a semantic `<footer>` element within the shell component. Styling uses existing design tokens from `src/styles/tokens.css`. The component authoring pattern (pure functions with `document.createElement`) is preserved. Tests follow the `tests/us-{number}/` convention. No runtime dependencies added. All content is static client-side text per trust rules.

**No breaking changes** to existing code, tests, build pipeline, or deployment.

### Consequences

- **Positive:** Makes the project's privacy guarantees explicitly visible to users at all times, fulfilling the credibility requirement
- **Positive:** Each trust statement is individually identifiable in the DOM, enabling targeted testing and potential future linking to detailed explanations
- **Positive:** Zero new dependencies keeps the bundle lightweight per CLAUDE.md constraints
- **Positive:** No regression risk to existing US-004 or US-005 tests (verified — no content assertions exist)
- **Negative:** None significant — this is a minimal, content-focused change to an existing component with no architectural risk
- **Trade-off:** Four inline statements require responsive wrapping consideration on narrow viewports; mitigated by flexbox layout with wrap
